== 実装済み基盤: Issue 001から026

実装済み基盤には次が含まれる。

* losslessなUTF-8ソース範囲、行索引、診断
* 段落、タイトル、セクション、constrained／unconstrainedな強調、斜体、等幅
* リテラルブロックと限定的なソースブロック
* HTML変換、Lint、保守的なFormatter、CLIコマンド
* Language Serverの診断、シンボル、アクション、Formatting、hover、補完
* 資源上限と、未対応構文に対する厳格・寛容ポリシー
* 出力形式に依存しないsemantic AST、Core 0.2規範文法、置換モデル
* Zed開発用拡張

Linux版Zedでの初期受け入れ確認は2026-07-20に完了した。LSP文書シンボルをOutlineへ
表示するには、ZedのAsciiDoc言語設定で`document_symbols`を`on`にする。

== AdocWeave内のホスト非依存統合: Issue 035から040

この段階では次を導入する。

* 汎用参照グラフと検索テキストのprojection（Issue 035完了）
* ホストが利用できる実行・キャッシュ・キャンセル境界（Issue 036完了）
* WASMとWeb Workerの統合（Issue 037完了）
* 定義移動、参照検索、文書リンクを含む残りのLanguage Serverプロトコル機能（Issue 038完了）

Language Serverの機能拡張に先立ち、Issue 045で手書きJSON-RPC基盤を
`async-lsp`へ移行し、`adocweave-lsp`の実行runtimeとしてTokioを採用する。
文書更新通知は受信順に適用し、読み取り要求は所有スナップショット上で有界に並行実行する。
`async-lsp`、TokioおよびLSP型はアダプター内に限定し、コア解析APIへ依存させない。
実行モデルと移行時の不変条件はxref:language-server.adoc[Language Server設計]に記載する。

コアはファイルシステム、ネットワーク、時刻、環境変数、UUID解決へ暗黙にアクセスしない。
ホストがこれらの能力を明示的に提供する。

ノートメタデータの型検証、DB投影、アプリ用サーバおよび製品統合は外部リポジトリが
所有する。

== 完了した中期ゴール: インライン解析と置換互換性: Issue 047から051

この中期ゴールでは、構文追加が集中しているインライン解析をrecognizerとsemantic builderへ
分離し、その基盤上で一般的なバックスラッシュエスケープ、unconstrained
書式、複数行spanを順に実装することである。各段階で規範文法、置換表、HTML、Formatter、
LSPおよびnative／WASM適合性を縦断して更新する。

一般的なバックスラッシュエスケープはIssue 048で実装を完了した。対応済み書式、属性参照、
URL、xref、STEM macroおよびanchorをnormal contextでliteral化し、不透明contextは変更しない。

Unconstrained書式はIssue 049で実装を完了した。strong、emphasisおよびmonospaceの二重
マーカーを境界非依存で認識し、Unicode境界、入れ子、隣接pair、空pairおよびescapeの
決定的な回復規則を固定した。

複数行spanはIssue 050で実装を完了した。段落を一つのinline列へ再構成し、LF／CRLFを
保持したまま書式とmacro labelを行境界越しに解析する。公開Paragraph API、projectionおよび
conformance snapshotはそれぞれversionを更新した。

Issue 047は初回検証後の批判的レビューで再開した。候補位置の前進だけでは線形時間を証明
できず、不正候補が密集した入力ではsuffix再探索が二次時間になり得るためである。不完全macroを
別走査で推測する構成を廃止し、complete、incompleteおよびinvalidを一つの走査器で分類した。
また、適合性schemaを公開ASTの`Serialize`実装からversion付きDTOへ分離し、同一Rustコアを
共有するnative／WASM比較とは独立した固定goldenを追加した。これらの是正を完了し、Issue 047は
完了した。

文書corpusの自己検査はIssue 051で導入した。既存の`docs-check`は`.md`混入検査だけという
契約を維持し、自己解析は別の`adoc-check` taskとしてverifyとrelease gateへ追加した。

== ドキュメント保守規則

リポジトリのドキュメントはAsciiDoc（`.adoc`）で記述し、`.md`ファイルを受け入れない。
構文を変更するたびに次を更新する。

* xref:syntax-support.adoc[対応表]
* xref:compatibility.adoc[外部仕様との互換性評価]
* xref:grammar.adoc[規範文法]
* 該当する場合はxref:substitutions.adoc[置換モデル]
* 対応するfixtureとテスト

`docs-check`は`.md`ファイルの混入だけを検査する。`adoc-check`は追跡対象の全`.adoc`を
決定的なpath順でAdocWeave自身へ渡し、寛容modeの診断をpath付きで報告する。解析失敗、
入力上限違反またはpanicによる異常終了はverifyとrelease gateを失敗させる。
