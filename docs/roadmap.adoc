= AsciiLoomロードマップ

このロードマップは予定する能力をまとめたものであり、リリース日を約束するものではない。
正式なタスク定義はxref:../issues/README.adoc[Issue計画]に記載する。

== 実装済み基盤: Issue 001から026

実装済み基盤には次が含まれる。

* losslessなUTF-8ソース範囲、行索引、診断
* 段落、タイトル、セクション、constrainedな強調、斜体、等幅
* リテラルブロックと限定的なソースブロック
* HTML変換、Lint、保守的なFormatter、CLIコマンド
* Language Serverの診断、シンボル、アクション、Formatting、hover、補完
* 資源上限と、未対応構文に対する厳格・寛容ポリシー
* 出力形式に依存しないsemantic AST、Core 0.1規範文法、置換モデル
* Zed開発用拡張

現在の開発環境にはZedのGUIセッションがないため、Zed実機の受け入れ確認は手動リリース
ゲートとして残る。

== 公開コアとレンダリング契約: Issue 027から028

次のアーキテクチャ段階では次を安定化する。

* 構文プロファイルとキャンセル可能な公開処理API
* 明示的なHTMLレンダリングポリシーと出力契約
* nativeおよびWASMホスト向けのバックエンド非依存処理結果

== ノート互換構文: Issue 029から034

この段階では依存順に次を追加する。

. 文書属性と保護メタデータ
. 明示アンカーと安定ID
. URLセキュリティポリシー付き通常リンク
. ブロック継続を含む順序付き・順序なしリスト
. ホスト提供Resolver境界の背後にあるノート参照マクロ
. 拡張境界の背後にあるSTEMおよびLaTeXノード

バックスラッシュエスケープ、unconstrained書式、passthrough、その他のAsciiDoc置換動作は
仕様上の未対応項目であり、この段階と合わせてIssue化する必要がある。より広いAsciiDoc
互換性を主張する前に規範文法へ追加しなければならない。

== ホスト統合: Issue 035から038

この段階では次を導入する。

* メタデータ、参照、検索テキストのprojection
* サーバー側実行、キャッシュ、キャンセル、観測性
* WASMとWeb Workerの統合
* 残りのLanguage Serverプロトコル機能

コアはファイルシステム、ネットワーク、時刻、環境変数、UUID解決へ暗黙にアクセスしない。
ホストがこれらの能力を明示的に提供する。

== 適合性とプロファイル公開: Issue 039から041

最後の予定段階では次を追加する。

* fuzzing、property test、セキュリティ回帰テスト
* nativeとWASMで共有する適合性fixture
* ノートアプリ互換プロファイルのリリース検証

== ドキュメント保守規則

リポジトリのドキュメントはAsciiDoc（`.adoc`）で記述し、`.md`ファイルを受け入れない。
構文を変更するたびに次を更新する。

* xref:syntax-support.adoc[対応表]
* xref:compatibility.adoc[外部仕様との互換性評価]
* xref:grammar.adoc[規範文法]
* 該当する場合はxref:substitutions.adoc[置換モデル]
* 対応するfixtureとテスト

AsciiLoomがリポジトリのドキュメントプロファイルを解析できるようになった段階で、
`docs-check`はAsciiLoom自身を使って`.adoc`内容を検証する。それまでは新しい`.md`
ファイルの混入だけを防止する。
