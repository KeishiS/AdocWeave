= アーキテクチャ

AdocWeaveはソース認識、意味解釈、エディターサービス、出力バックエンドを分離する。

[source,text]
----
ソーステキスト
  -> 損失のないソース行とCST
  -> 出力形式に依存しないsemantic AST
  -> 一回の意味解析
       -> 診断
       -> ReferenceTarget索引
       -> 未解決Reference一覧
       -> 文書サービス、Formatter
       -> HTMLバックエンド
       -> 将来の出力バックエンド

CLIとLanguage ServerはコアAPIのアダプターである。
----

semantic ASTはHTMLではなく言語上の意味を表現する。インライン内容では通常テキスト、
再解釈しないリテラル内容、スタイル付き子要素を区別する。ブロックノードはソース範囲と
意味値を保持するが、タグ、class、レンダリングオプションを保持しない。HTMLモジュールは
ASTへ依存するが、Parser、AST、Lint、Formatter、CLI、Language ServerはHTMLモジュールへ
依存しない。

コアはソースと設定を明示的に受け取り、ファイル、ネットワーク、環境変数、時刻、
外部コマンドへアクセスしない。これらの副作用はホストアダプターが所有する。

== 単一解析パイプライン

公開 `parse_document()` は字句・構文解析を一回だけ行い、そのASTをLintと文書索引へ
渡す。Lintの各ルール、HTMLレンダラー、CLIおよびLanguage Serverが同じソースを
個別に再解析してはならない。

`ParseResult` はCST、AST、診断、参照先索引および未解決参照を同じ文書版の
スナップショットとして返す。Language Serverの文書状態はこの診断とASTを保持し、
publish diagnosticsやCode Actionの要求ごとに解析し直さない。

`CORE_API_VERSION` は公開コア型のフィールド、意味またはエラー契約に破壊的変更を
加えたときに更新する。リリース前で後方互換性を保証しない期間でも、ホスト側が
コンパイル時定数を照合して不一致を検出できるよう、この規則を省略しない。

単独利用向けの `lint(source, config)` は便利関数であり、内部で一回だけ解析して
`lint_parsed(source, ast, config)` へ委譲する。既にASTを持つ呼び出し側は
`lint_parsed` を使用する。

== 出力契約の所有権

HTML契約バージョンは入力ポリシーではなく、生成物の性質である。このため
`RenderPolicy` に選択されても検証されないバージョン値を持たせず、
`HtmlOutput.contract_version` として返す。未実装機能の設定項目は先行して公開せず、
実際に意味を持つIssueで追加する。
