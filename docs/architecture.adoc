= アーキテクチャ

AdocWeaveは標準AsciiDocの解析コア、出力バックエンド、ホストアダプターおよび
アプリケーション固有拡張を分離する。

[source,text]
----
ソーステキスト
  -> Engine::analyze
       -> 所有型Analysis
            ├─ SourceDocument
            │    ├─ lossless token
            │    ├─ 行・改行索引
            │    └─ UTF-8／UTF-16位置変換
            ├─ lossless syntax tree
            ├─ semantic AST
            ├─ Diagnostics
            ├─ ReferenceTarget
            └─ ASTから導出するsemantic query
                 ├─ Formatter
                 ├─ HTMLその他の出力バックエンド
                 ├─ 汎用文書projection
                 ├─ CLI
                 ├─ Language Server
                 └─ ホスト固有の解析・参照解決
----

== 単一解析スナップショット

`Analysis`は入力テキストと行索引を一元化した`SourceDocument`、syntax tree、AST、診断および参照先を
所有する。reference、macroおよびresource queryはASTを唯一の正本として導出し、別の深い複製索引を
保持しない。同じ`Analysis`に含まれる値は、すべて同じ文書版と`ParseOptions`から
生成される。

CLI、Formatter、HTMLバックエンドおよびLanguage Serverは、既存の`Analysis`を
再利用しなければならない。各機能がソースを再解析したり、独自の行索引を構築したり
してはならない。文字列だけを受け取る便利関数は、内部で`Engine::analyze`を一度だけ
呼び出してよい。

`CORE_PROFILE_VERSION`は構文と既定意味、`CORE_API_VERSION`は公開型とAPI、HTML生成物は
`HTML_CONTRACT_VERSION`で管理する。各versionと再構築条件は
xref:core-profile.adoc[AdocWeave Core Profile 8]で定義する。

== Native host境界

`adocweave-host` crateは`adocweave-cli`とLanguage Serverが共有するnative filesystem境界である。canonical path、
許可root、regular file、UTF-8、file数、resource byte数、総byte数および依存graphのforward／reverse closureを
ここで一度だけ実装する。`adocweave` core packageはこのcrateに依存せず、I/Oを行わない。CLI packageとLSP adapterだけが
host crateを利用し、検証済みtextからcoreの`ResourceSnapshot`を構築する。

LSPの解析jobはrootの`DocumentRevision`と`WorkspaceGeneration`を同時に取得する。採用時にどちらかが変化して
いれば、直接解析とinclude展開解析を含むjob全体を破棄する。採用済み結果は一つの`DocumentView`がroot解析と
任意の展開解析を所有し、機能ごとに異なる世代の結果を合成しない。

== 構文層と意味層

syntax treeは元入力をバイト単位で再構築でき、空白、改行、不完全構文および未対応領域を保持する。
semantic ASTはHTMLタグ、class、出力設定を持たず、AsciiDoc上の意味だけを表現する。
出力バックエンドはsemantic ASTを各形式へ写像し、ソース文字列を再解析しない。

構造nodeはblockに加え、見出しmarker、block属性とdelimiter、list itemとmarker、inline spanと
delimiter、macro、target、label、errorおよびunknownを表す。triviaと未注釈部分は共有する
`SourceDocument`のlossless token leafとして保持する。syntax treeは増分解析を保証しない。

`Paragraph`はblock全体の`range`と、末尾改行を除く`content_range`、元のLF／CRLFを含む
`value`および一つの`inlines`を所有する。行単位のsemantic nodeは持たず、
行とoffsetの対応は`SourceDocument`と各inlineのsource rangeから導出する。

`AstBlock::range()`を全ブロック範囲の標準入口とする。共通の不変走査はsemantic walkerを
利用する。属性展開だけはlowering中のcrate内部可変走査で行う。出力固有処理はenumを明示的かつ
網羅的に分岐し、新しいvariantを黙って無視するwildcard分岐を避ける。

すべてのsemantic blockは出力非依存な`BlockMetadata`を所有し、`AstBlock::metadata()`を
共通query入口とする。title、ID、role、optionおよびelement attributeは値とsource rangeを
保持する。syntax treeは元のメタデータ行と順序を保持し、HTML rendererは未知属性を入力由来の
HTML属性へ直接写像しない。

区切り付きblockは`DelimitedBlockKind`と`DelimitedContent`を直交させ、contentを
`Compound(Vec<AstBlock>)`、`Verbatim`、`Raw`または`Table`として型付けする。`Raw`は信頼済みHTMLを
意味せず、入力由来の危険な値をbackendが見落とさないための型である。delimiter判定と閉じ規則は
一つのregistryに集約し、compound再帰は`max_block_depth`を超えてはならない。

インライン解析は、次候補をsource位置順に選ぶ走査制御、markerまたはmacroの範囲と回復位置を
返すrecognizer、および認識結果からsemantic `Inline`を作るbuilderを分離する。recognizerは
ASTを構築せず、builderは開始・終了記号を再探索しない。不完全入力を認識できない場合も、
再開位置は次のUTF-8境界へ必ず前進しなければならない。

block parserは`recognize_line`の固定優先順位表で行を分類し、`BlockCursor`へ型付きの
`BlockRecognition`をcommitする。commitは現在位置より後ろかつ文書範囲内への遷移だけを受理する。
syntax nodeとsemantic blockの追加およびpending metadataの付与は共通commit処理を用いる。
recognizerが生成した回復factはsyntax issueへ移動した後に破棄し、最終semantic ASTからqueryできない。
doctype依存のheading正規化はfactの移動前に行い、構文診断と意味判断の順序を固定する。

== 標準コアとホスト拡張

標準コアは文書属性を名前、原値、操作および範囲として保持するが、`note-id`、タグ、
日時その他のアプリケーション固有型を知らない。未知のxrefスキームも
`ReferenceDestination::Scheme`として損失なく保持する。

ホストは標準解析の完了後に、同じ`Analysis`からアプリケーション固有の値を検証・投影
する。DB、ファイル、ネットワーク、時刻、環境変数およびアクセス制御はホストが所有する。
ホスト診断は標準診断と別の結果として生成し、必要なら表示層で決定的に統合する。

検索テキスト、参照edge、通常リンクおよびアンカー索引は、同じ`Analysis`を入力とする
xref:projections.adoc[汎用文書projection]から取得する。ノートアプリ固有projectionは
その出力を別リポジトリで変換し、コアASTへDBスキーマを追加しない。

Parserへ任意コードを注入するプラグイン機構は提供しない。新しい表面構文には、
AdocWeave側でsyntax tree、AST、回復規則、診断およびfixtureを定義する必要がある。

== 純粋性と信頼境界

コア解析と出力バックエンドは、明示的な入力と設定だけから決定的な結果を返す。
外部I/Oや参照解決を解析中に実行しない。Resolverが返すhrefも信頼済みHTMLではなく、
HTML出力前に必ず`UrlPolicy`で検証する。

`ProcessingLimits`は入力、行、ブロック、属性、参照、catalog entry・投影byte数、リスト深さ・継続数、インライン深さ、
属性展開、数式および診断の上限を一元管理する。上限超過時は診断または公開エラーへ変換し、
深い入力を上限超過後もそのまま再帰処理してはならない。

Issue 052以降、入力、行および深さ上限に加え、block、semantic node、属性、参照および診断の
個数上限を構築中に検査する。構造要素は共有ParseBudgetを追加前に消費し、診断は上限到達後に
新しい値を構築しない。解析完了後の個数検査を資源防御の代わりにしてはならない。

== Language Server実行境界

Language Serverのprotocol処理には`async-lsp`、実行runtimeにはTokioを使用する。
これらは`adocweave-lsp`のアダプター依存であり、`adocweave`コアの公開API、解析型および
ホスト実行契約へ持ち込まない。

文書更新通知はmain loopで受信順に`DocumentStore`へ適用する。読み取り要求は開始時に
文書versionと所有スナップショットを取得し、その後は文書状態を借用せずに並行実行する。
CPU解析はSemaphoreで投入数を制限してから`spawn_blocking`へ渡す。Futureの破棄だけでは
実行中の同期解析を停止できないため、LSPキャンセルをコアの`CancellationCheck`へ接続する。

解析結果を公開する直前に文書versionまたはgenerationを再確認する。結果が古い場合、
診断、補完、編集その他の応答へ採用しない。lockを使用する場合も、CPU解析または
client通信のawait中に保持してはならない。

通知順序、backpressure、キャンセルおよび検証項目の詳細は
xref:language-server.adoc[Language Server設計]で定義する。

Language Server以外のサーバ、Workerおよびデスクトップホストも
xref:server-execution.adoc[ホスト実行・キャッシュ契約]に従い、同じrevision採用判定と
本文を含まない観測モデルを使用する。

ブラウザ統合のversion付きschema、協調キャンセルおよびCSP要件は
xref:wasm-worker.adoc[WASM APIとWeb Worker統合]に定義する。

== 持続可能性の不変条件

M9完了時の設計レビューにより、今後の変更では次を維持する。

* parserをCLI、HTML、LSP、WASMまたはホストごとに複製せず、一つの`Analysis`を再利用する。
* ASTへブロックまたはインラインvariantを追加した場合、visitor、projection、HTML、
  Formatter、Lint、LSPおよび規範fixtureの網羅性を同じ変更で確認する。
* 公開schemaは`CORE_API_VERSION`、`HTML_CONTRACT_VERSION`、
  `CORE_PROFILE_VERSION`、`PROJECTION_CONTRACT_VERSION`、
  `CONFORMANCE_CONTRACT_VERSION`または`WASM_API_VERSION`の
  対応する契約で管理する。
* async runtime、LSP型、WASM bindingおよび製品DB型をコアcrateへ持ち込まない。
* キャッシュキーは`ParseOptions`を網羅的に分解し、設定追加時にコンパイルエラーで
  更新漏れを検出する。
* 外部参照索引はversionとgenerationを検証し、完全でない索引からRenameを返さない。
* 通常の観測情報に文書本文、属性値、URL、xref locatorまたは生成HTMLを含めない。

大きなmoduleの行数だけを理由に層を増やさない。分割時は、構文意味、出力、protocol変換、
実行制御またはホストI/Oという責務境界に沿い、同じASTを別表現へ複製しない。

Issue 052から058の再設計では、追加で次を目標不変条件とする。

* `Analysis`とsemantic ASTは構築完了後に変更できず、公開APIから不整合なsnapshotを作れない。
* source表記はsyntax rangeまたはtokenを正本とし、正規化・置換後の値だけを所有する。
* 構文診断、lowering診断およびsemantic・style lintを区別し、ASTへ診断変換専用の状態を複製しない。
* 共通walkerは木の再帰だけを共有し、出力backend固有の意味mappingを隠さない。

Issue 063後の横断レビューでは、さらに次を不変条件とする。

* blockのHTML変換は再帰的な単一dispatcherを通す。top-level、compound containerおよび
  list continuationごとに`AstBlock`のmatchを複製しない。新variantの出力漏れは
  compilerの網羅性検査で検出する。
* inline認識の事前indexも走査予算へ含める。各delimiterの次位置を一回の逆走査で構築し、
  openerごとの残り文字列検索を禁止する。大量の未閉じmarkerでも入力長に対して線形に保つ。
* well-formedかつ正しい位置のdocument titleだけが著者・改訂metadataの認識を開始できる。
  malformed headingからheader状態を推測しない。
* table、標準macroおよびpreprocessorは`parser.rs`へ全処理を追記せず、責務別moduleに
  recognizer、builderおよび上限検査を置く。`parser.rs`はline dispatchと構成を担当する。
* `AstBlock`または`Inline`をwildcardで黙って捨てるconsumerを作らない。意図的に処理しない
  variantも列挙し、walker、lowering、projection、Lint、Formatter、LSPおよびbackendを監査する。

Issue 066後の横断レビューでは、Issue 001からの実装を再確認し、次を追加した。

* preprocessorのroot文書とinclude文書は単一の行状態機械を共有する。directive追加時に条件評価、
  escape、上限およびsource map処理を二重実装しない。
* resource取得はhost、決定的な展開はコアという境界を維持する。コアへfilesystem、HTTP client、
  async runtimeまたは環境依存のpath canonicalizationを導入しない。
* `parser.rs`と`inline.rs`は依然として最大の変更集中点である。今後の構文追加では既存の巨大な
  dispatchへ条件分岐を足す前に、table、macro、preprocessorと同様の責務moduleへrecognizerと
  semantic builderを置く。ただし行数削減だけを目的とした薄いwrapper分割は行わない。
* WASM DTOでコア型を再定義する場合は、同一snapshotによるcross-runtime testと契約version更新を
  同じ変更単位に含める。

Issue 067ではこの方針を実装へ反映し、block／inline semantic model、syntax node builderおよび
syntax diagnostic projectionを解析engineから分離した。`parser`と`inline`は公開facadeとして単一型を
再公開するが、互換用の旧型、変換処理または二重pipelineは持たない。

inline scanner、delimiter index、macro／marker recognizerおよびbuilderは、候補index、回復cursorと
走査budgetを共有する一つの線形engineとして維持する。private tokenをmodule間APIへ露出するだけの
物理分割は行わない。block固有parserも同様に、独立した不変条件を持つ場合だけ専用moduleへ分離する。

この再設計では旧新モデルを同時維持する移行layerを設けず、全adapter、runtime、fixtureおよび
文書を新しい単一pipelineへ収束させた。
