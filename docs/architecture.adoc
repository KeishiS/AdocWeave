= アーキテクチャ

AdocWeaveは標準AsciiDocの解析コア、出力バックエンド、ホストアダプターおよび
アプリケーション固有拡張を分離する。

[source,text]
----
ソーステキスト
  -> Engine::analyze
       -> 所有型Analysis
            ├─ lossless CST
            ├─ semantic AST
            ├─ LineIndex
            ├─ Diagnostics
            ├─ ReferenceTarget
            └─ Reference
                 ├─ Formatter
                 ├─ HTMLその他の出力バックエンド
                 ├─ 汎用文書projection
                 ├─ CLI
                 ├─ Language Server
                 └─ ホスト固有の解析・参照解決
----

== 単一解析スナップショット

`Analysis`は入力テキストを含むCST、AST、行索引、診断、参照先および未解決参照を
所有する。同じ`Analysis`に含まれる値は、すべて同じ文書版と`ParseOptions`から
生成される。

CLI、Formatter、HTMLバックエンドおよびLanguage Serverは、既存の`Analysis`を
再利用しなければならない。各機能がソースを再解析したり、独自の行索引を構築したり
してはならない。文字列だけを受け取る便利関数は、内部で`Engine::analyze`を一度だけ
呼び出してよい。

`CORE_API_VERSION`は公開型、意味、診断またはエラー契約の破壊的変更時に更新する。
HTML生成物の変更は別に`HTML_CONTRACT_VERSION`で管理する。

== 構文層と意味層

CSTは元入力をバイト単位で再構築でき、空白、改行、不完全構文および未対応領域を保持する。
semantic ASTはHTMLタグ、class、出力設定を持たず、AsciiDoc上の意味だけを表現する。
出力バックエンドはsemantic ASTを各形式へ写像し、ソース文字列を再解析しない。

`AstBlock::range()`を全ブロック範囲の標準入口とする。インライン列の横断処理は
`AstDocument::visit_inline_sequences()`と可変版を利用する。ブロック階層の意味を
必要とする処理はenumを明示的かつ網羅的に分岐し、新しいブロックを黙って無視する
wildcard分岐を避ける。

== 標準コアとホスト拡張

標準コアは文書属性を名前、原値、操作および範囲として保持するが、`note-id`、タグ、
日時その他のアプリケーション固有型を知らない。未知のxrefスキームも
`ReferenceDestination::Scheme`として損失なく保持する。

ホストは標準解析の完了後に、同じ`Analysis`からアプリケーション固有の値を検証・投影
する。DB、ファイル、ネットワーク、時刻、環境変数およびアクセス制御はホストが所有する。
ホスト診断は標準診断と別の結果として生成し、必要なら表示層で決定的に統合する。

検索テキスト、参照edge、通常リンクおよびアンカー索引は、同じ`Analysis`を入力とする
xref:projections.adoc[汎用文書projection]から取得する。ノートアプリ固有projectionは
その出力を別リポジトリで変換し、コアASTへDBスキーマを追加しない。

Parserへ任意コードを注入するプラグイン機構は提供しない。新しい表面構文には、
AdocWeave側でCST、AST、回復規則、診断およびfixtureを定義する必要がある。

== 純粋性と信頼境界

コア解析と出力バックエンドは、明示的な入力と設定だけから決定的な結果を返す。
外部I/Oや参照解決を解析中に実行しない。Resolverが返すhrefも信頼済みHTMLではなく、
HTML出力前に必ず`UrlPolicy`で検証する。

`ProcessingLimits`は入力、行、ブロック、属性、参照、リスト深さ、インライン深さ、
数式および診断の上限を一元管理する。上限超過時は診断または公開エラーへ変換し、
深い入力を上限超過後もそのまま再帰処理してはならない。

== Language Server実行境界

Language Serverのprotocol処理には`async-lsp`、実行runtimeにはTokioを使用する。
これらは`adocweave-lsp`のアダプター依存であり、`adocweave`コアの公開API、解析型および
ホスト実行契約へ持ち込まない。

文書更新通知はmain loopで受信順に`DocumentStore`へ適用する。読み取り要求は開始時に
文書versionと所有スナップショットを取得し、その後は文書状態を借用せずに並行実行する。
CPU解析はSemaphoreで投入数を制限してから`spawn_blocking`へ渡す。Futureの破棄だけでは
実行中の同期解析を停止できないため、LSPキャンセルをコアの`CancellationCheck`へ接続する。

解析結果を公開する直前に文書versionまたはgenerationを再確認する。結果が古い場合、
診断、補完、編集その他の応答へ採用しない。lockを使用する場合も、CPU解析または
client通信のawait中に保持してはならない。

通知順序、backpressure、キャンセルおよび検証項目の詳細は
xref:language-server.adoc[Language Server設計]で定義する。

Language Server以外のサーバ、Workerおよびデスクトップホストも
xref:server-execution.adoc[ホスト実行・キャッシュ契約]に従い、同じrevision採用判定と
本文を含まない観測モデルを使用する。
