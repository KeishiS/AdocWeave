= WebAssemblyとWeb Worker

`adocweave-wasm`は、ブラウザーやJavaScriptからAdocWeaveを呼び出すための入口です。HTML、
問題一覧、見出しなどは、一度の解析結果から作られます。`VERSION`は`0.6.4`です。

Web Workerを使うと、変換処理を画面を描く処理と分けられます。Workerのメッセージ形式は内部用です。
アプリケーションからはxref:browser-library.adoc[ブラウザーで使う]で案内している公開モジュールを使ってください。

== ビルド

[source,console]
----
cargo make build-wasm
cargo make wasm-size
----

生成物は追跡対象外の`target/adocweave-wasm/`へ出力する。`wasm-size`はWASM本体とJS glueの
byte数を表示する。release判断時は同じtoolchainとprofileで計測し、値をCI artifactへ
保存する。初期ロード時間とpeak memoryはブラウザ・端末依存であるため、製品ホスト側で
Performance APIとブラウザのmemory profilerを用いて測定する。

2026-07-20時点で計測した旧基準値は、WASM本体503,638 bytes、
JS glue 17,290 bytesである。canonical AST導入前の489,038 bytesからの増加は、derived
`Debug`を廃止し、完全なASTをcanonical JSONとして直列化する処理の導入による。この値は
機能追加時の比較基準であり、恒久的な上限ではない。

== API schema

入力は未知フィールドを拒否する。

[source,json]
----
{
  "packageVersion": "0.6.4",
  "sourceId": "host:opaque-document",
  "version": 1,
  "generation": 18,
  "source": "= Title\n",
  "products": {
    "html": true,
    "diagnostics": true,
    "projection": true
  },
  "renderInputs": {
    "references": [],
    "resources": []
  },
  "options": {
    "syntaxMode": "permissive"
  }
}
----

`options`はnativeの`ParseOptions`に対するoverrideであり、省略したフィールドにはコアの
既定値を使う。これによりJSへ既定上限を複製しない。profile、処理上限、保護属性および
URL policyを明示すればruntime間で同じ設定になる。未知フィールドは拒否する。
`limits`には`ProcessingLimits`の各フィールドをcamelCaseで必要なものだけ指定できる。
異なるpackage versionは`unsupported-api-version`として拒否され、応答には`expected 0.6.4`を含める。

`renderInputs`はxref:render-inputs.adoc[解決済みrenderer入力契約]のWASM表現であり、
`references`と`resources`の各要素へ`sourceStart`、`sourceEnd`およびtagged `outcome`を指定する。
成功outcomeは`{"status":"resolved","href":"...","displayText":"公開タイトル"}`、失敗outcomeは
`{"status":"failed","kind":"missing"}`の形を取る。失敗outcomeは安定kindだけを保持し、
自由形式messageを受理しない。resource成功値には
任意の`mediaType`と`byteLength`を追加できる。参照の`displayText`は省略可能なプレーンテキストで、
空labelかつ解決成功の場合だけ使用する。不正なrangeは`invalid-render-input`として拒否する。

応答は入力version、generation、`packageVersion`、parse概要および要求したproductを返す。`products`はsyntax、
canonical AST、HTML、`attributeOccurrences`、diagnostics、symbolsおよびprojectionを選択する。省略時はbrowser向け
既定値（HTML、diagnostics、projection）を使用する。選択しないproductは空の文字列・配列または`null`で返る。
`attributeOccurrences`はsource順の標準文書属性を返し、重複、set／unset、空文字値および`range`、`nameRange`、
`valueRange`を保持する。projection自身も文字列の`packageVersion`を持つ。エラーは`code`と`message`だけからなる。

[source,json]
----
{
  "code": "unsupported-api-version",
  "message": "unsupported package version 0.5.0 (expected 0.6.4)"
}
----

JSへ渡すversionとgenerationは`u32`に制限し、JavaScript numberの整数精度差を避ける。
ソース、応答およびエラーは呼び出しごとの所有値であり、次回呼び出しで無効化される
借用pointerを公開しない。

== Worker protocol

UIはxref:browser-library.adoc[`AdocWeaveClient`]を通じて専用Workerへ要求する。
clientはgenerationを単調増加させ、共有された32 bit整数へ最新generationを書いてから
要求を送る。Workerは短いdebounce後に最新要求だけを実行し、versionとgenerationが
現在値に一致する結果だけを通知する。

同期WASMの実行中、Workerのmessage loopは次のキャンセル通知を処理できない。そのため、
`SharedArrayBuffer`上のgenerationを`Atomics.load`するcallbackを
`CancellationCheck`へ接続する。main threadはWASM実行中でも`Atomics.store`でき、
Parserの検査点で協調停止できる。

`SharedArrayBuffer`を利用できない環境では、公開clientが古いWorkerを`terminate()`して
新しいWorkerを生成する。この差は公開APIの内側へ隠す。Futureの破棄やmessage送信だけで同期WASMが
停止したと仮定してはならない。

== 資源上限と縮退

WASMもネイティブと同じ`ProcessingLimits::default()`を使用する。既定では10 MiBを超える
入力を解析前に`limit-exceeded`として拒否する。大きな文書で部分的な別parserへ切り替えず、
最後の成功結果をUIに保持したまま、ホストが上限エラーと再試行手段を表示する。

応答は要求したproductだけを直列化する。必要な出力は`products`で明示し、不要な大きな
canonical ASTや属性occurrenceを要求しない。product集合を変更する場合も、requestの`packageVersion`は
対象releaseの値と完全一致させる。

== Cross-runtime適合性

`cargo make test-cross-runtime`は同じJSON要求をnative adapterと実際のwasm32成果物へ渡し、要求した
syntax tree、AST、診断、シンボル、projectionおよびHTMLを比較する。fixture manifestは
xref:../fixtures/conformance/cases.json[`fixtures/conformance/cases.json`]である。通常の
`cargo make verify`にもこの試験を含める。

== CSPと信頼境界

SharedArrayBufferにはcross-origin isolationが必要であり、通常は次のheaderをmain documentと
Worker moduleを含む同一contextの全応答へ設定する。

[source,text]
----
Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Embedder-Policy: require-corp
----

CSPでは少なくとも`worker-src 'self'`を設定し、WASMの読み込み元を`script-src`と
`connect-src`で必要最小限に許可する。ブラウザが要求する場合だけ
`'wasm-unsafe-eval'`を追加し、`unsafe-eval`全体を許可しない。

生成HTMLは信頼済みDOMではない。ホストは`innerHTML`へ無条件に挿入せず、AdocWeaveの
HTML契約とホスト側sanitization・表示sandboxを適用する。KaTeX等の数式レンダラーは
別の依存・CSP・入力検証境界として統合する。
