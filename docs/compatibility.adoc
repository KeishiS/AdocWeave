= AsciiDoc仕様との互換性

== 参照基準

AdocWeaveは次の公式資料を外部仕様の参照基準とする。

* https://asciidoc.org/[AsciiDoc公式サイト]
* https://docs.asciidoctor.org/asciidoc/latest/[AsciiDoc Language documentation]
* https://docs.asciidoctor.org/asciidoc/latest/_exports/asciidoc-pre-spec.pdf[AsciiDoc pre-specification, 2026-02-03版]
* https://docs.asciidoctor.org/asciidoc/latest/syntax-quick-reference/[AsciiDoc syntax quick reference]

AsciiDocプロジェクトは言語仕様が現在も策定中であると説明している。そのためAdocWeaveは、
未確定の最終仕様への準拠を主張せず、採用した参照基準と意図的な差異を記録する。

Core 0.4は意図的に小さく定義したプロファイルである。そのテスト成功が証明するのは
xref:grammar.adoc[AdocWeaveの規範文法]への準拠であり、AsciiDoc全体への準拠ではない。

== 書式ペアの互換性

公式言語文書はconstrainedとunconstrainedの書式ペアを区別する。constrainedペアは
書式マーカーに隣接する文字へ依存する。unconstrainedペアは二重マーカーを使い、
単語内部やCJK文字に隣接する箇所にも書式を適用できる。

参照資料:

* https://docs.asciidoctor.org/asciidoc/latest/text/[Text formatting and punctuation]
* https://docs.asciidoctor.org/asciidoc/latest/text/troubleshoot-unconstrained-formatting/[Troubleshoot unconstrained formatting pairs]
* https://docs.asciidoctor.org/asciidoc/latest/text/monospace/[Monospace]

[cols="2,2,2",options="header"]
|===
|機能 |AsciiDoc参照動作 |AdocWeave Core 0.4

|Constrained strong
|単一の`*`ペア
|対応

|Constrained emphasis
|単一の`_`ペア
|対応

|Constrained monospace
|単一のバッククォートペア
|対応

|Unconstrained strong
|二重の`**`ペア
|対応

|Unconstrained emphasis
|二重の`__`ペア
|対応

|Unconstrained monospace
|二重のバッククォートペア
|対応

|連続行をまたぐ書式
|書式ペアは連続する複数行をまたげる
|対応。LF／CRLFを保持した段落inline列として解析する

|その他のテキストスタイル
|ハイライト、下付き、上付き、組版用句読点を定義する
|未対応
|===

AdocWeaveのconstrained境界条件は、現時点ではCore 0.4固有の明示的契約である。
識別子中の記号を誤認しないよう設計しているが、策定中のAsciiDoc言語に含まれる全境界例と
等価であることはまだ証明していない。

Unconstrained形式は単語、CJK、emojiおよび句読点の境界に依存せず認識する。同種の
マーカーが連続するrunは左から2文字ずつpairへ分割する。この規則により、隣接する
`**one****two**`は二つのstrong spanになる。空pairは通常テキストとして保持する。
この隣接runと不完全pairの回復規則はAdocWeaveの決定的な部分集合契約であり、AsciiDocの
すべてのprocessorにおける曖昧な入れ子結果との同一性までは保証しない。

== バックスラッシュエスケープ

公式文書は、通常内容で置換を抑止する方法の一つとしてバックスラッシュを定義する。
バックスラッシュは書式、属性参照、文字置換、アンカー、マクロ、URL自動認識などを抑止でき、
レンダリング結果からは除去される。

参照資料:
https://docs.asciidoctor.org/asciidoc/latest/subs/prevent/[Escape and prevent substitutions]

.参照仕様の動作
[source,asciidoc]
....
\*Stars* remains literal markup rather than strong text.
\{name} remains an attribute reference in source form.
\https://example.org is not activated as a link.
....

AdocWeaveは対応済みのconstrained書式、属性参照、通常リンク、`xref:`、`<<...>>`、STEM macro
およびanchor開始記号でこの動作へ対応する。直前のescape用バックスラッシュは認識を抑止し、
表示から除去する。リテラル、source、等幅およびSTEM内容は不透明であり、その内部の
バックスラッシュを変更しない。未対応構文に対する予防的escapeは解釈しない。

実装時には単にバックスラッシュを削除するだけでなく、次を満たす必要がある。

* lossless syntax treeがエスケープ文字とその範囲を保持する。
* semantic textがエスケープ済みマークアップをliteral textとして保持する。
* 診断とFormatterが著者の意図を保持する。
* 置換コンテキストごとにエスケープの有効性を定義する。
* HTMLおよび将来のバックエンドがソースを再解析せず意味内容を利用する。

バックスラッシュエスケープはレンダラー固有の置換としてではなく、一般的な置換モデルと
一体で実装する。

== 置換の互換性

公式文書は、特殊文字、quotes、属性、文字置換、マクロ、後置換という順序付きの置換種別を
説明している。また`normal`、`header`、`verbatim`、`pass`、`none`などの置換グループを
定義している。

参照資料:

* https://docs.asciidoctor.org/asciidoc/latest/subs/[Substitutions]
* https://docs.asciidoctor.org/asciidoc/latest/subs/special-characters/[Special character substitutions]

AdocWeaveは現在、次の意味上の対応を実装する。

* 段落と見出しでconstrained／unconstrainedな強調、斜体、等幅を解析する。
* 等幅、リテラルブロック、ソース本体を不透明として扱う。
* 出力バックエンドが出力先で危険な文字をエスケープする。
* 文書属性を再帰展開し、循環・深さ・生成byte数を制限する。
* URL、`link:`、`xref:`、`<<...>>`およびLaTeX STEMを限定的に解析する。
* 一般的な文字置換、highlight、subscript、superscript、typographic quoteおよび
  安全なpassthrough意味ノードを扱う。

コンテキストごとの置換段階は型で固定している。任意の`subs`属性による段階の追加・削除と、
未対応macro固有の置換は今後の範囲である。実装済み契約は
xref:substitutions.adoc[Core置換モデル]で定義する。

Markdown風見出し、fenced code blockおよびMarkdown blockquoteは公式quick referenceにも掲載されるが、
AsciiDoc言語のoptionalなAsciidoctor機能と明記されている。AdocWeaveはこれらを標準profileの欠落とは
数えず、標準構文の実装後に独立した互換optionとして検討する。

== 互換性方針

機能が実装されるまでは次を守る。

* ソースを黙って破棄しない。
* 未対応内容を実行可能なraw HTMLへ変換しない。
* 寛容モードでは回復可能な内容を保持する。
* 厳格モードでは、認識した未対応ブロック構文を全コア操作で拒否する。
* 出力バックエンドがソーステキストを再解析して意味を推測しない。

今後の構文Issueでは、この文書、対応表、規範文法、バックエンド横断fixtureを同じ変更で
更新しなければならない。
