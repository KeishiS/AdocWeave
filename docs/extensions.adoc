= 独自機能を追加する

別のアプリケーションでAdocWeaveを使い、独自のリンク、メタデータ、検索などを追加する方法を説明します。
AdocWeave本体は標準AsciiDocを読みます。ノートアプリ固有のデータベースや権限、画面の作りは
利用側アプリで扱います。

== 二段階解析

ホスト拡張は、必ず標準コア解析の後に動作する。

[source,text]
----
let analysis = Engine::new(options).analyze(source)?;
let host_result = host_extension.analyze(&analysis, host_context).await?;
----

ホストは`Analysis`を変更せず、別の結果型へアプリケーション固有の診断、メタデータ投影、
解決済み参照および検索情報を格納する。標準診断とホスト診断を統合する場合は、範囲、
診断コードおよび安定IDによる決定的な順序を定義する。

== 属性拡張

ホストは`Analysis.ast.attributes`の名前、`raw_value`、操作および範囲を使用する。
たとえばノートアプリは`note-id`をUUIDとして検証できるが、コアASTへノート専用variantを
追加してはならない。

属性検証は次を満たす。

* 同じ入力、設定およびホストスキーマから同じ結果を返す。
* 診断を属性全体ではなく、可能な限り`name_range`または`value_range`へ付ける。
* 未知属性を暗黙に削除または書き換えない。
* 正規化値と原値を区別し、Formatterが原値を失わないようにする。
* DB、時刻等を必要とする検証は純粋な構文・型検証と分離する。

== xrefスキーム拡張

ノート参照には独立マクロを追加せず、標準xrefを使用する。

[source,asciidoc]
----
xref:note:123e4567-e89b-12d3-a456-426614174000#section[関連ノート]
----

Parserは未知のスキームも`ReferenceDestination::Scheme`へ保持する。ホストは
`Analysis::reference_queries()`から`ReferenceKey::Scheme`を取得し、scheme、locator、
anchorを検証・解決する。標準ParserやLintへUUID規則を登録してはならない。

構文解析中にResolverを呼ばない。不完全xrefにも標準解析結果を返し、I/O待ちによって
Parser、Formatterまたは標準Lintを停止させない。

== Resolver

I/Oを必要とするホストは`ReferenceResolver`を実装する。アプリ専用メソッドではなく、
`resolve_scheme(scheme, locator)`で不透明な`SourceId`を返し、`resolve_anchor()`で
アンカー確認と最終href生成を行う。

Resolverは次を守る。

* `Send + Sync`であり、複数文書から同時に利用できる。
* キャンセルされても共有状態を破損しない。
* 対象不在、アンカー不在、曖昧、ルート外、権限不足および内部障害を安定コードへ正規化する。
* OSパス、DB主キー、認証情報を診断やhrefへ漏らさない。
* 参照元`SourceId`、文書versionおよび`TextRange`を失わない。

== キャッシュと無効化

キャッシュキーには参照元`SourceId`、正規化済み`ReferenceKey`、構文プロファイルversion、
参照元文書versionおよびホストスキーマversionを含める。対象文書、アンカー索引、権限
またはスキーマが変化した場合、成功結果と失敗結果をともに無効化する。

== エディター統合

補完、Hover、Definition、ReferencesおよびDocument Linkは同じ`Analysis`、
`ReferenceQuery`および解決結果を共有する。機能ごとに対象文字列を再解析しない。
入力途中の構文失敗と、確定した対象不在を区別する。

LSPアダプターはxref:language-server.adoc[Language Server設計]の通知順序、generation、
キャンセルおよびlock規則に従う。`ReferenceResolver`の契約自体は特定async runtimeへ
依存せず、外部ホストへTokioを要求しない。

== HTMLと信頼境界

Resolverが返すhrefは信頼済みHTMLではない。HTML出力前にAdocWeaveの`UrlPolicy`を
必ず通過させる。拡張はraw HTML、イベント属性、`javascript:`または許可されていない
`data:` URIを許可する経路を提供してはならない。

URL検査では文書入力とResolver出力のcontextを混同しない。たとえばhostは文書入力の
`/notes/id`を拒否したまま、Resolverが生成した同じroot-relative URLだけを許可できる。外部リンク属性、
source言語、数式言語、未解決参照表示およびresource capabilityは`RenderPolicy`へ設定し、HTML文字列の
置換で実装しない。

== 互換性

ホストは起動時またはビルド時に`CONTRACT_VERSION`を照合する。公開型またはResolver契約の
破壊的変更に備え、Rust enumの序数を永続化せず、未知の値を安全な未解決状態として扱う。
