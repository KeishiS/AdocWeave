= 継続的インテグレーション

この文書は、repositoryの変更検証とGitHub Release候補の受け入れを分離するCI契約を定義する。実行内容の正本は
`.github/workflows`と`Makefile.toml`であり、この文書はeventごとの保証と運用境界を説明する。

== Event model

[cols="2,3,3,2",options="header"]
|===
|Event |常に実行する検証 |追加する配布検証 |外部変更

|pull request
|source quality、fuzz、Nix package、`dist plan`
|なし。target別archiveは生成しない
|なし

|`main` push
|source quality、fuzz、`dist plan`
|Linux 2 target、global archive、metadata、導入E2E
|一時Actions artifactだけ

|version tag
|source quality、fuzz、tag／version
|mainと同じcandidateに加えてprovenance
|承認後にGitHub Releaseを原子的に公開
|===

pull requestの更新では同じrefの古いrunをcancelする。mainとtagはcandidate byte列および公開処理を途中で置き換えないため
cancelしない。全jobには用途に応じたtimeoutを設け、中間build artifactは7日、集約candidateは14日で期限切れにする。

== Quality lane

`quality.yml`はcallerのeventを参照しない。callerはpublication tagだけを明示入力し、空の場合はtag検証を行わない。
qualityは次の独立jobを持つ。

Dependency governance::
固定RustSec revision、root／Zedのlicense・source・advisory、重複version、noticeおよび全lockfile境界を検査する。

Source quality::
Chromiumを含まないNixの`ci` shellで`cargo make quality`を実行する。format、Clippy、全test、WASM、Zed、
TypeScript、cross-runtime、docsおよびrelease契約を含む。

Fuzz quality::
`fuzz-check`とshort fuzzを独立jobで実行する。source qualityと並列化するが、publishは両方の成功を必要とする。

Nix package quality::
pull requestではNix package、runtime closureおよびNixOS評価を独立jobで実行する。mainとtagでは、同じ検証を
Linux x86-64／ARM64のcandidate installation jobへ移し、同じsource buildを重複させない。

Linuxの開発者向けdefault shellは任意の実browser受け入れ用にNix Chromiumを引き続き含む。aarch64 Darwinでは
対応packageがないため含めない。現行nixpkgsが対応を終了したx86-64 Darwin shellは公開しない。CIでは実browserを起動せず、
巨大なChromium closureの取得失敗やrunner固有のsandbox差によってRust、CLI、LSPおよび配布契約の検証まで停止することを
避ける。Rust toolchain、cargo-distおよび検査toolは`ci` shellでも`flake.lock`に固定する。

Rust toolchainは`flake.lock`で固定したcurrent stableだけを使う。root workspaceとZedの`rust-version`、
release manifestの`rustVersion`およびNix package buildは同じversionに揃える。古いcompilerの互換性検証は行わない。

== Candidate lane

mainとtagだけがcandidate laneへ入る。native archiveは対応architectureのrunnerでbuildし、別jobへ渡したbyte列を同じ
architectureで起動する。browserとZed archiveは`release-global-artifacts`で、実際にuploadするarchiveへ決定性、型、size
およびlayout試験を適用する。集約後はmanifest、SBOM、checksumとasset集合を再生成比較し、さらに
Linux 2 architectureでclean HOMEへの導入・完全削除E2Eを実行する。このE2Eは抽出したbrowser archive内の
WASMを直接実行し、解決済み表示テキストとkind-only failure projectionも検査する。同じnative runnerで
flake packageをsource buildし、CLI／LSP version、runtime closureおよびNixOS package設定を検査する。

実Chromiumによるisolated／fallback contextとbundler経路は`cargo make browser-runtime-check`へ分離する。この任意gateは
browser機能を優先する時点または公開後の手動受け入れで実行し、現行のquality／candidate／publishを阻害しない。

cargo-distはnetwork installerを直接実行せず、`flake.lock`で固定したNix closureから`run-pinned-dist.sh`を介して起動する。
このwrapperはNix store由来と`dist-workspace.toml`のversionを実行直前に検証し、planと両native buildで共有する。

candidate buildはsource qualityと並列に開始する。publishはquality、fuzz、candidate verificationおよびinstallation E2Eの
すべてを待つため、待ち時間を短縮しても公開判断の検証範囲は変えない。

== Publish lane

publishはversion tag、全quality jobおよびcandidate laneの成功を必要とする。tag commitは現在の`origin/main`と完全一致
しなければならない。write permissionとOIDCはprotected `github-release` Environment内の再利用workflowだけへ与える。
private draftのasset集合とattestationを確認し、最後にdraftを解除する。失敗時は同じrelease IDのdraftだけを削除する。

== Repository設定

branch protectionまたはrulesetでは、少なくともqualityのsource quality、fuzz quality、Nix package qualityとplanを
pull requestの必須checkにする。
dependency governanceも必須checkとし、RustSec取得障害を一時的にskipしてmergeしてはならない。
workflow名やjob名を変更するとGitHub上のcheck名も変わるため、CI再構成をmergeした直後にrequired checksを確認する。
`github-release` Environmentはversion tagだけを許可し、必要なreviewerを設定する。

workflow policyは`yq`でYAML objectを構築し、jobごとの条件、依存edge、権限、Environment、timeout、artifact retentionおよび
step所属を検査する。`actionlint`はschemaとGitHub式の検査を担当する。コメントや別jobに同じ文字列を置いてもpolicyを
満たしたことにはならない。

== 失敗の切り分け

plan:: release train、cargo-dist設定、tag選択またはtarget matrixを確認する。
dependencies:: RustSec revision、例外期限、license、source、重複baseline、noticeまたはlockfile境界を確認する。
Rust toolchain:: `flake.lock`、release manifestおよびroot／Zedの`rust-version`が一致することを確認する。
quality:: 同名の`cargo make` taskをlocal Nix shellで再実行する。
native smoke:: archive名、ELF architecture、static linkまたはLSP handshakeを確認する。
global artifacts:: browser size／TypeScriptまたはZed archive layoutを確認する。
candidate verification:: 6 archiveの欠落、未知file、digest、SBOMまたはsource commitを確認する。
installation E2E:: runner architecture、version選択、PATHまたはuninstall後の残存fileを確認する。
publish:: Environment承認、既存release、asset集合、attestationまたはdraft cleanupを確認する。

失敗したtag runのassetを手動で既存Releaseへ追加してはならない。修正はmainでqualityとcandidateを通し、新しいstable
version tagで最初から実行する。
