semantic ASTでは`Text`、`Literal`、`Styled`、`AttributeReference`、`Link`、
`Reference`および`Formula`がこの区別を表現する。

段落は行ごとの置換列ではなく、LF／CRLFを含む一つのinline列を所有する。書式spanと
リンク・xrefの表示文字列は連続行をまたげる。出力バックエンドはnormal inline contextの
改行を一つの表示上の空白へ写像し、不透明なblock本文の改行規則と混同してはならない。

- `Text`は出力先バックエンドが解釈する通常テキストである。
- `Literal`は不透明な内容であり、再帰的な置換を行わない。
- `Styled`は再帰的に解析した意味上の子要素を所有する。
- `Link`はURLターゲットと意味解析済み表示文字列を分離する。
- `Reference`は参照先表記と意味解析済み表示文字列を分離し、解決をホストへ委譲する。

normal contextでは、対応済み構文の開始記号直前にあるescape用バックスラッシュを一つ除去し、
対象全体を`Text`として保持する。verbatim、literal、等幅および数式contextではescape置換を
適用せず、バックスラッシュを内容の一部として保持する。
- `Formula`は言語と不透明な数式本文を保持し、AsciiDoc置換を再帰適用しない。

出力バックエンドはこのASTだけを入力とし、ソースマークアップを再解析してはならない。
HTMLはテキストを含む全コンテキストで`&`、`<`、`>`、`"`、`'`をエスケープする。
将来の非HTMLバックエンドも、解析動作を変更せず出力形式に対応する安全規則を適用する。

名前に`substitutions`を含むテストがコンテキスト横断の動作を固定する。

リンクターゲットの属性展開はsemantic ASTの構築時に一度だけ行う。Lintと
出力バックエンドは、いずれも同じ展開済みターゲットをURLポリシーへ渡す。
xrefターゲットはコアで属性展開せず、ホストが返した解決済みhrefを出力時に検査する。

== 将来の置換コンテキストの追加

後続Issueでリストまたは数式等を追加する場合、構文ノードと同じ変更で
この表へ行を追加しなければならない。各行は少なくとも、インライン解析、属性展開、
raw出力の可否およびバックエンド側の安全化責任を定義する。

置換済みの意味構造はsemantic ASTへ格納する。HTMLその他のバックエンドが元ソースを
再解析したり、独自に属性展開したりしてはならない。不透明コンテキストは`Literal`または
対応するブロックノードとして保持し、子要素の置換を禁止する。
