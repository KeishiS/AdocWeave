= Language Server設計

この文書は`adocweave-lsp`の責務、実行モデルおよびIssue 045以降の設計を定義する。
AsciiDocの構文・意味・診断・整形は`adocweave`コアが所有し、Language Serverは
LSPとの変換、文書世代管理および実行制御だけを所有する。

== 現在の状態と移行範囲

現在の実行バイナリは`async-lsp`のstdio、ライフサイクルおよび型付きRouterを使用し、
診断、Outline、Code Action、Format、Hover、補完、Definition、References、
Document Link、条件付きRenameおよびSemantic Tokensを提供する。意味処理は
runtime非依存の型付き`LanguageService`へ分離され、手書きJSON dispatchとstdio framingは
削除済みである。
Linux版Zedでの初期受け入れ確認は移行前の実装で完了している。`async-lsp`移行とIssue 038の
機能追加後の回帰確認は別途必要である。

Issue 045で、既存機能の意味処理とfixtureを維持しながら、protocol基盤を
https://github.com/oxalica/async-lsp[`async-lsp`]、実行runtimeを
https://tokio.rs/[Tokio]へ破壊的に移行した。この文書は現在の実装契約を表す。

== crate境界

[source,text]
----
Zedまたは他のLSP client
  -> async-lsp main loop
       -> protocol adapter
            -> DocumentStore
            -> LSP意味処理service
                 -> adocweave::Engine
                      -> 所有型Analysis
----

`adocweave-lsp`は次を所有する。

* LSPのinitialize、shutdown、exitおよびstdio transport
* URI、position encodingおよびLSP型とコア型の変換
* 開いている文書とworkspace resourceのtext、version、generationおよび解析スナップショット
* file URIに限定したinclude依存graph、逆依存索引および上限付きlocal resource読込
* 通知順序、要求並行数、CPU解析数、debounceおよび結果採用判定
* clientへの診断、応答、通知および進捗報告

`adocweave`コアは次を所有する。

* AsciiDocのsyntax tree、AST、診断、Formatterおよびprojection
* UTF-8ソース範囲とスナップショット固有の`SourceDocument`
* `ProcessingLimits`と協調的な`CancellationCheck`
* 外部I/Oを行わない決定的な`Engine::analyze`

`async-lsp`、TokioおよびLSP型をコアcrateの依存関係または公開APIへ持ち込んではならない。
Issue 036のホスト実行契約も特定async runtimeへ依存しない。

== 通知と要求

LSP通知と要求は同じ並行処理規則を使用しない。

`didOpen`、`didChange`、`didSave`および`didClose`はmain loopで受信順に適用する。
source、versionおよびgenerationはhandler内で更新し、CPU解析だけを有界workerへ渡す。
状態更新を別taskへ移してはならない。

文書ごとに待機中または実行中の解析taskを高々一つ保持する。新しい更新は前taskのtokenを
発火してtask handleをabortし、debounce待機taskが入力回数に比例して蓄積しないようにする。
`spawn_blocking`開始後はabortだけに依存せず、コアの協調キャンセルで停止する。

文書同期はincrementalを広告する。一つの`didChange`に複数変更がある場合、各rangeを
その直前の変更適用後の本文と交渉済みposition encodingへ順番に適用する。rangeなしの変更は
その時点で全文を置き換える。LFとCRLFを独自に数えず、コア`SourceDocument`の行索引を使用する。

読み取り要求は並行処理できる。handlerは開始時に次を所有値として取得し、その後は
`DocumentStore`を借用し続けない。

* 文書URI
* clientが送った文書version
* server内部のgeneration
* ソースまたは既存`Analysis`のスナップショット
* position encodingと有効な設定

この分離により、通知は決定的な順序で状態を変更し、HoverやDocument Symbol等の要求は
同じ時点のスナップショット上で安全に並行実行できる。

`DocumentStore`はcopy-on-writeの所有mapを使う。要求開始時の`LanguageService` cloneは
文書数に比例してURIやmapを複製せず、更新時だけmapを分岐する。外部公開APIから
`DocumentStore`を直接変更させず、`run`と`run_with_host_index`がgeneration不変条件を
所有する。

== CPU解析とbackpressure

`Engine::analyze`は同期CPU処理である。Tokio executor上で直接実行せず、CPU解析用
`Semaphore`を取得してから`spawn_blocking`へ渡す。

同時LSP要求数と同時CPU解析数は異なる上限として設定する。Tokioのblocking thread上限は
他のblocking処理と共有され、待ちqueueにも適切なbackpressureを与えないため、
AdocWeaveの解析上限として使用しない。

Semaphore待機中に要求がキャンセルされた場合は、解析を開始しない。permitは解析完了または
失敗時に必ず解放し、panic時にも文書状態のlockを残さない。

読み取り要求もhandler開始時点の所有スナップショットをFutureへ移し、同じSemaphoreで
上限を設けた`spawn_blocking`ワーカー上で計算する。したがって、main loopや
`DocumentStore`の借用中にCPU処理を行わない。`ConcurrencyLayer`による要求Future数の上限と
SemaphoreによるCPU処理数の上限は別の制約である。

== キャンセルと古い結果

`$/cancelRequest`によってLSP要求のFutureが破棄されても、開始済みの`spawn_blocking`処理は
自動停止しない。要求ごとのdrop guardを`CancellationCheck`へ接続し、CPUワーカーは開始前、
対応機能の検査点および完了後に協調的に停止または結果を破棄する。

文書解析は要求固有tokenとは別にgeneration固有tokenを持つ。`didChange`、`didClose`、
shutdownおよびexitは該当tokenを発火し、解析内部の既存検査点で停止させる。

結果をclientへ公開する直前に、対象文書が開かれており、取得時のgenerationと現在の
generationが一致することを再確認する。一致しない解析から次を公開してはならない。

* 診断
* Hover、補完、シンボルその他の応答
* Code ActionまたはWorkspace Edit
* 参照解決結果またはDocument Link

キャンセルとgeneration不一致は通常の構文診断ではなく、結果を採用しないための
実行制御である。

読み取り要求も取得時の文書tokenを保持する。要求計算中に`didChange`または`didClose`が
tokenを発火した場合、計算済みであっても応答を`ContentModified`として破棄する。これにより
古いFormatting Edit、Code Action、Definitionその他の応答を更新後に送信しない。

== lock規則

共有状態のlock区間は、文書の検索、スナップショット取得、generation比較および
結果差し替えに限定する。次の処理中にlockを保持してはならない。

* `Engine::analyze`その他のCPU処理
* ResolverのI/O
* Semaphoreの待機
* clientへのrequest、notificationまたは進捗送信

文書単位の更新順序はmain loopで保証し、文書状態全体を長時間lockすることで代用しない。

== async-lsp middleware

採用するmiddlewareはコード上で明示し、既定featureだけに依存しない。

* server lifecycle
* 要求の同時実行制限と`$/cancelRequest`
* panic隔離
* 本文を記録しない構造化tracing

client process監視、network transportその他の不要なfeatureは、利用要件が生じるまで
有効にしない。LSP仕様型はprotocol境界でのみ使用し、コア型との変換関数をテスト可能な
安定面とする。

== 診断

Language Serverは直接解析の`Analysis.diagnostics`と、include展開解析の型付きorigin projectionを
LSP診断へ変換し、同じ入力を独自にLintしない。
push diagnosticsを送る場合、文書URI、versionまたはgenerationおよびposition encodingを
同じスナップショットから得る。

ホストResolver由来の診断は標準診断と区別し、安定した順序で統合する。Resolver障害が
コア構文診断の公開を妨げてはならない。

push diagnosticsを継続する。文書解析完了eventで現在generationを再確認してから通知し、
pull diagnosticsは現段階では広告しない。

== workspace設定と参照索引

`workspace/didChangeConfiguration`は直下または`adocweave`オブジェクト内の
`debounceMs`を受け付け、0から1,000 msへ制限する。不正設定は現在値を維持し、サーバを
終了させない。既定は30 msである。

既定サーバはinitialize時のworkspace folderから`.adoc` fileを上限付きで読み、open文書を優先する
version付きsnapshotを構築する。`include::`の依存graphと逆依存索引を保持し、open文書変更または
`workspace/didChangeWatchedFiles`で変更resourceに依存するrootだけを新generationとして再解析する。
symlinkは追跡せず、workspace外、network URI、1 resource 10 MiB超、合計50 MiB超または10,000 file超を
拒否する。LSP自身はnetwork resourceを取得しない。

include展開結果は直接解析と別のimmutable snapshotとして採用し、diagnostic、include Definition、
Document Link、ReferencesおよびRenameを元`SourceId`とrangeへ投影する。欠落resource、循環、safe
mode違反および上限超過は`adocweave-project`診断となる。古いresource versionを含むjobはroot文書の
generation gateで破棄する。

任意スキームおよび製品DBを扱うホストは`run_with_host_index`へ`HostReferenceIndex`を注入する。
このtraitは事前構築済みのメモリ索引を問い合わせる境界であり、実装内で同期I/Oを行っては
ならない。

`HostReferenceRequest`は参照元URI、文書version、generation、`ReferenceKey`およびposition
encodingを所有する。ホストは要求generationより古い索引からLocationを返さず、対象不在・
曖昧・索引更新中は`None`、ホスト障害は`Err`として区別する。障害を受けた要求は失敗しても、
コア診断、Outline、Formatその他の機能とサーバプロセスは継続する。

Renameはhost indexが結果を返す場合はそれを優先し、それ以外は採用済みworkspace snapshot内の宣言と
参照だけを編集する。対象がworkspace外へ解決される場合や参照集合を構築できない場合は編集を返さない。

== エディター投影

Definition、ReferencesおよびDocument Linkは文書内、相対文書および任意スキームを同じ
`ReferenceKey`で扱う。通常の外部URLは`UrlPolicy`を通したDocument Linkとして別に扱う。
Semantic Tokensは見出しmarker・本文、通常リンク・xrefの参照先およびアンカーIDを
`Analysis`から生成し、ソースを再解析しない。

Hoverは見出し、文書属性、アンカー、通常リンク、xref、属性参照および数式の意味を表示する。
補完はsource block言語と、現在文書で解析済みのアンカー候補を提供する。URLや数式本文を
推測補完せず、任意スキーム・未open文書の候補はホスト側索引の将来拡張領域とする。

== 検証

移行では既存機能の回帰試験に加え、少なくとも次を自動検証する。

* initialize前、重複initialize、shutdown後およびexitのライフサイクル
* `didOpen`から複数`didChange`までの受信順
* 更新通知直後の要求が更新前の状態を観測しないこと
* 異なる文書の要求を並行処理できること
* LSP要求数とCPU解析数の独立した上限
* Semaphore待機中とCPU解析中のキャンセル
* 古いgenerationの診断と編集を破棄すること
* UTF-8、UTF-16およびCRLFの位置変換
* stdioとin-memory transportで同じprotocol fixtureを利用できること

自動試験後にZedで診断、Outline、Code Action、Format、Hover、補完、Definition、
References、Document LinkおよびSemantic Tokensを再確認する。
