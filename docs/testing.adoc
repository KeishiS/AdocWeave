= 品質保証とFuzzing

この文書は、AdocWeaveの決定的試験、fuzzingおよび不具合回帰fixtureの運用契約を定義する。

== 試験層

通常の`cargo make verify`では、workspaceの単体・統合試験の一部として次を毎回実行する。

* `cargo make test-properties`: 生成したUTF-8入力について、CST復元、範囲、位置変換、
  Formatter、HTML、projectionおよび参照索引の不変条件を検査する。
* `cargo make test-security`: 攻撃的入力、危険なURL、Resolverが返したURL、処理上限、
  不正UTF-8および協調的キャンセルを検査する。

`cargo make fuzz-check`はすべてのfuzz targetをコンパイルする。`cargo make fuzz-smoke`は
開発時またはCIで短時間探索する。`FUZZ_SECONDS`を指定しない場合はtargetごとに5秒実行する。
smoke testは`target/`下へseedを複製し、追記された探索corpusを終了時に破棄するため、
リポジトリのseed corpusを変更しない。
CI用の`cargo make ci`は`verify`に加えて`fuzz-check`と`fuzz-smoke`を実行する。

== リポジトリ文書corpus

`cargo make docs-check`は`.md`混入だけを検査する。`cargo make adoc-check`はGitが追跡する
すべての`.adoc`をbyte順のpathで列挙し、既定処理上限と寛容modeでAdocWeave自身へ渡す。
未追跡ファイル、`target/`生成物およびネットワークには依存しない。

寛容modeの診断は成功時にも`[path]`、行、列、severityおよびcode付きで
`target/adoc-check-diagnostics.log`へ完全に残し、標準出力へ件数と保存先を表示する。
これは未対応記法を含む仕様例と意図的な異常fixtureをcorpusから除外せず分類するためである。
診断の存在だけでは失敗にせず、解析失敗、入力上限違反、位置不正またはpanicによる非0終了を
文書pathと詳細付きで即時表示し、全fileの検査後に失敗させる。構文対応が広がった段階で
診断baselineを縮小し、将来のstrictな文書subsetは別taskとして追加する。

現在の`adoc-check`は文書corpusに対する非crash・公開エラー検査であり、文書が診断なしで
妥当であることを証明しない。Issue 058ではこの役割を`adoc-corpus-check`相当として明文化し、
通常の規範文書に予期しない診断がないことを検査する`docs-lint`と分離する。意図的な異常
fixtureは期待診断付きmanifestまたはgoldenへ分類し、通常文書のbaselineに混ぜない。

[source,console]
----
cargo make fuzz-check
FUZZ_SECONDS=30 cargo make fuzz-smoke
----

長時間探索は通常の検証ゲートから分離し、targetを明示して実行する。

[source,console]
----
adocweave-fuzz run --fuzz-dir fuzz analyze -- -max_total_time=3600
adocweave-fuzz run --fuzz-dir fuzz process -- -max_total_time=3600
adocweave-fuzz run --fuzz-dir fuzz render -- -max_total_time=3600
----

`adocweave-fuzz`はflakeで固定したnightly Rustと`cargo-fuzz`だけを選択するラッパーである。
通常のビルド、Clippyおよびテストは引き続きstable Rustを使用する。

== Fuzz targetと公開入力境界

[cols="1,2,3",options="header"]
|===
|Target |入力境界 |主な不変条件

|`analyze`
|`Engine::analyze`へ渡す任意UTF-8
|panicしない、CSTが入力を復元する、すべてのtoken範囲が正しいUTF-8境界にある

|`process`
|`process`、`process_check`へ渡す任意バイト列
|全operationがpanicせず、UTF-8・上限違反を型付きエラーとして返す

|`render`
|解析済みsemantic ASTを受け取るHTML backend
|同じ入力で同じ出力を返し、危険なURLを有効なリンクとして生成しない
|===

位置変換、Formatter、URL分類、xref正規化、projectionおよびResolver境界は、入力をsemantic
ASTへ限定した関数であるため、決定的property testとsecurity testで検査する。WASMとLSPの
transport境界はIssue 040の共通fixture runnerで追加検査する。

Issue 040で、xref:../fixtures/conformance/cases.json[共通conformance manifest]をnativeと
実WASMで実行する`cargo make test-cross-runtime`を追加した。各caseは互換性分類と差分理由を
必須とし、CST、AST、診断、シンボル、projectionおよびHTMLを応答全体で比較する。

`ConformanceSnapshot`は比較専用のversion付き契約である。CST token表現、semantic ASTの
構造・順序、診断JSON、シンボルJSON、projection JSONまたはHTMLの意味を変更する場合は、
fixtureの`contractImpact`を再判定し、必要なら`CONFORMANCE_CONTRACT_VERSION`を更新する。

AST適合性値はcompact JSONであり、Rustの`Debug`表示ではない。公開semantic APIから独立した
version付きconformance DTOを使用し、`schemaVersion`、意味node種別、UTF-8 byte range、
意味値、子nodeおよびsource順を明示的に固定する。AST型の変更と適合性schemaの変更を同一視せず、
DTO変換とgolden fixtureの差分をレビューしてversionを判定する。
Rustの型が同じであることだけをruntime一致の代わりにせず、必ず実wasm32成果物を実行する。
ただしnativeとWASMは同じ解析コアを共有するため、この比較だけを意味上の正しさの証拠に
してはならない。包括的caseはリポジトリに固定した期待AST、診断、projectionおよびHTMLとも
比較する。

`attributes-anchors-links-references-lists-stem` caseはAST、診断、projection、symbolおよびHTMLの
固定goldenを持つ。runnerの出力から試験時に期待値を再生成してはならない。意図した契約変更では
各差分をレビューし、`contractImpact`と契約versionを判定してから期待ファイルを更新する。

== Formatterの不変条件

Formatterは生成入力に対して次の三条件を満たさなければならない。

. 冪等性: `format(format(x)) = format(x)`
. 意味保存: 検索可能テキスト、参照先および参照キーが変化しない
. バイト保存: `PreserveBytes`と分類したCST領域へEditを生成しない

意味保存の定義を構文追加に合わせて拡張しなければならない。単に出力文字列が安定している
ことだけを意味保存の証拠としてはならない。

走査量の上限試験は、入力長から算出した推定値を実測値の代わりにしてはならない。Issue 052で
scannerとdelimiter indexが実際に検査した位置を計測し、密集した不正候補を含む入力に対して
定数倍の上限を検証する。

block、semantic node、属性および参照の個数上限は、上限直前、同値および1超過を公開Engineで
検証する。超過試験は最終ASTの個数だけでなく、構築中のParseBudgetが返すresource、limit、actualを
確認しなければならない。診断上限は全診断を構築してから切り捨てず、生成時に強制する。

== 不具合corpusの運用

`fuzz/corpus/<target>/`には探索開始用の小さいseedを保存する。cargo-fuzzが問題を発見した
場合は次の順序で処理する。

. `fuzz/artifacts/<target>/`の入力で問題を再現する。
. `cargo fuzz tmin`で入力を最小化する。
. 原因を修正し、最小入力を通常の回帰fixtureまたはテストへ追加する。
. 新しい構文経路を探索する価値がある場合だけ、最小入力をseed corpusにも追加する。

`fuzz/artifacts`とビルド生成物はGitへ登録しない。再発防止の正本は通常試験に置き、fuzzing
だけを実行しなければ検出できない状態を残さない。
