= tower-lsp-serverへのLSP基盤移行

== 目的

手書きJSON-RPC dispatchと部分的なLSP型定義を、保守中の`tower-lsp-server`へ移行し、
今後のキャンセル、並行処理、設定、参照機能およびsemantic tokensを安全に追加できる
基盤を作る。

元の`tower-lsp`ではなく、コミュニティフォーク
https://github.com/tower-lsp-community/tower-lsp-server[`tower-lsp-server`]を採用する。
採用時点の候補は0.23系列であり、リポジトリのMSRV 1.85と一致することを確認する。

== 実装範囲

* `LanguageServer`実装をtransport・protocol境界とし、意味処理を独立serviceへ委譲する。
* 現在の`DocumentStore`、`Analysis`、位置変換および機能テストを再利用する。
* stdio framing、JSON-RPC error、初期化、shutdown、exitおよびclient通知をライブラリへ委譲する。
* `ls-types`の型をLSP境界だけで使用し、コアParser、AST、診断へ持ち込まない。
* 文書状態を安全な共有所有権へ移し、lock保持中にCPU解析やclient awaitを行わない。
* CPU解析を有界workerで実行し、`$/cancelRequest`を`CancellationCheck`へ接続する。
* 文書versionを結果採用時に再確認し、古い解析結果を公開しない。
* push diagnosticsと既存capabilitiesを移行前後で維持する。
* 手書きtransport、dispatchおよび一時的なprotocol型を移行完了後に削除する。

== 完了条件

* 既存のLSPテストが型付きrequest・response経路で成功する。
* 不正JSON-RPC、initialize前request、重複initialize、shutdown後requestを安定して処理する。
* キャンセル済みまたは古いversionの診断・Editを送信しない。
* 同時request数とCPU解析数に明示的な上限がある。
* Zedで診断、Outline、Code Action、Format、Hoverおよび補完が従来どおり動作する。
* `adocweave-lsp`に手書きLSP kind番号とstdio framingが残らない。
