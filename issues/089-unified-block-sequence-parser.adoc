= Block sequence parserの再帰context統合
:status: completed
:depends-on: 075

== 目的

top-level文書、compound blockおよびAsciiDoc table cellが別々に持つline dispatchを、context差を型で表す
単一のblock sequence状態機械へ統合する。新しいblock種別を一か所へ追加すれば、すべての再帰contextへ
適用されるか、明示的に拒否されなければならない。

== 背景

Issue 072のcell parserは親budgetと個別recognizer／builderを共有するが、paragraph、listおよび区切り付きblock
だけを選ぶ独自loopである。このためdocument headerを禁止するという妥当なcontext差と、heading、literal
paragraph、break等の未接続を区別できず、将来の構文追加漏れを型検査で検出できない。

== 作業

* `BlockContext`へheader metadataの可否、headingの扱い、block／table深さおよび解析範囲を持たせる。
* top-levelと再帰contextを同じcursor、recognizer、dispatchおよびpending metadata状態機械で解析する。
* compound blockとAsciiDoc cellから旧`parse_compound_children`を削除する。
* budget、cancel、source rangeおよびorphan metadataの不変条件を一つの実装で維持する。
* top-levelで対応する各block variantについて、cell内での許可または明示的な診断をtable-driven testにする。
* parser moduleの責務境界、表仕様、文法、対応表およびconformance fixtureを同時更新する。

== 実装方針

`BlockContext`は少なくとも文書root、compound containerおよびAsciiDoc cellを区別し、document header、
document attribute、heading、再帰深さと解析終端のpolicyを値として持つ。認識優先順位そのものをcontextごとに
複製せず、認識後の許可・回復だけをpolicyで決める。

状態機械は`BlockSequenceOutput`としてlossless syntax node、lowering前のblock fact、明示anchorおよび
syntax issueを返す。再帰contextを一時的な`AstDocument`へloweringしてはならない。そうすると属性展開、
heading正規化、catalog／文書構造構築およびbudget計上が子ごとに再実行され、文書全体の正本が分断されるためである。
rootだけが全出力を集約した後にloweringとcatalog構築を一度実行する。

Issue 089の時点では`SourceDocument::from_fragment_bounded`をglobal byte offsetを維持する断片入力として
利用した。これは所有文字列と行索引を再構築するため、Issue 090でroot sourceを借用する`BlockInput`と
source-backed indexed viewへ置換した。内容変換済みcellだけが断片入力を使用する。

== 完了条件

再帰context専用のline dispatchが存在せず、全block variantのcontext policyを列挙したtestがある。PSVの
AsciiDoc cellで許可したheading、literal paragraph、break、source／math block、metadata付きblockおよび
入れ子tableが同じsource rangeと共有budgetで解析される。禁止するdocument header要素は通常paragraphへ
偶然fallbackせず、規範化した診断または意味へ決定的に回復する。

== 完了結果

文書root、compound containerおよびAsciiDoc cellを型付き`BlockContext`で区別し、単一の
`parse_block_sequence`へ統合した。状態機械は`BlockSequenceOutput`としてlowering前のsyntax、block、anchor、
attributeおよびheader factを返す。属性展開、heading正規化、catalog／文書構造構築とsyntax diagnostic投影は
rootで一度だけ実行し、旧`parse_compound_children`は削除した。

再帰contextは親のnode／block／table budget、block／table深さ、cancel checkおよびglobal UTF-8 byte rangeを
共有する。cell内の見出し、literal paragraph、break、source／STEM、metadata、明示anchor、syntax diagnosticと
入れ子表をtestおよびnative／WASM conformance caseで固定した。明示anchorのtarget探索とHTML ID生成も共通walkerで
全階層へ拡張した。カスタムseparatorの表delimiterを規範文法へ追加し、Core Profile、HTMLおよびconformance契約を
更新した。

Issue 094では単一loopを維持したまま出力をroot／nestedへ型分離した。nested syntaxはcompound blockとAsciiDoc
table cellの親syntax nodeへ接続し、document headerと属性factはroot stateだけが生成する。
