= Release導入文書とend-to-end受け入れ
:status: in-progress
:depends-on: 086,087

== 目的

GitHub Releaseから初めて導入する利用者とfrontend integratorが、source checkoutなしでassetを選択、検証、
install、更新および削除できることを最終的に確認する。

== 作業

* OS／architectureから正しいCLI・LSP assetを選ぶ表と、checksum／attestation検証手順を追加する。
* user-local install先、system-wide install、PATH、shell completion、更新、version固定およびuninstallを
  platform別に記載する。`curl | sh`を唯一の手順にしない。
* browser libraryの静的hosting、ES module／WASM MIME type、CSP、cross-origin isolation、cache busting、
  integrity確認およびリアルタイムHTML previewの利用例をfrontend向け文書へ追加する。
* Zed Gallery導入、binary cache、PATH override、offline動作および完全削除を開発版手順と分離する。
* GitHub prereleaseをclean Linux x86-64、Linux ARM64、browser frontendおよびLinux版Zed profileへ導入する
  E2E checklistを実行する。macOS／Windowsは将来targetの受け入れIssueへ分離する。
* release noteに対応target、契約version、breaking change、既知の制約、asset verificationおよびupgrade手順を
  必須化する。
* 配布元はリポジトリ所有者自身のGitHub Releasesだけとする。package manager、container registry、
  crates.io、npmおよびZed Registryへの公開は行わず、導入経路もGitHub Releasesだけで完結させる。
* private repositoryで検証する場合は、認証済みGitHub CLIまたは最小権限tokenを使う手順と、credentialを
  archive、logおよび設定例へ残さないことを確認する。

== 完了条件

source checkoutとRust toolchainのないclean環境でCLI、LSP、browser libraryおよびZed連携を確認し、install前後の
filesystem差分からuninstall後に管理対象fileが残らないことを確認する。最終手順を
`docs/release-checklist.adoc`と利用者向け導入文書へ反映する。

== 実装状況

* xref:../docs/release-installation.adoc[利用者向け導入文書]へasset選択、checksum／attestation、user-local／
  system-wide配置、PATH、version固定、rollback、browser hosting、Zed、更新および削除手順を集約した。
* `release-installation-e2e.mjs`は隔離HOMEへCLI、LSP、browser libraryおよびZed拡張を導入し、clean PATH、
  version symlink、公開入口およびuninstall後のfile集合一致を検査する。
* Galleryへ公開しない方針とsource checkout不要の完了条件を両立するため、version付き
  `adocweave-zed-<version>.tar.xz`を6番目のarchiveとして配布契約へ追加した。
* 公開済み`v0.1.0-rc.1`を変更せず、新しいasset契約と公開契約version 2をrelease candidateへ分離した。
  失敗した`v0.1.0-rc.2` tagも付け替えず、Chromium終了競合の修正を含む`v0.1.0-rc.3`を受け入れ対象とする。
* GitHub Release noteへ対応target、公開契約とbreaking change、既知制約、asset検証および
  upgrade／rollbackを必ず追加する生成・検証処理をpublish境界へ追加した。
* local x86-64成果物では、Zed archiveの決定性検査と導入・完全削除E2Eが成功した。
* Release workflowは集約candidateをx86-64／ARM64のnative runnerへ戻し、両architectureの導入・完全削除E2Eが
  成功するまでpublish jobへ進まない。
* pull requestは完全local quality gateとplan、`main`へのpushはさらに非公開candidate検証を起動し、tag作成前にmain
  commitの成功結果を確認できるようにした。通常のmain pushではpublish jobを起動しない。

== 残る受け入れ

`v0.1.0-rc.3`を公開した後、source checkoutとRust toolchainを持たないclean Linux x86-64／ARM64で実Release assetを
使うE2Eを実行する。browserは実配備header、Zedは初回managed LSP取得、offline再起動およびUIからの削除までを
確認する。この公開と実機操作はrepository所有者のtag作成、Environment承認および対応環境を必要とする。
