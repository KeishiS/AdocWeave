= CI bootstrap固定とworkflow構造検証
:status: completed
:depends-on: 102

== 目的

CIが実行前に取得するtool自体の完全性を固定し、GitHub Actions policyをコメントや無関係な文字列では満たせない
構造検証へ移行する。

== 監査結果

従来はcargo-dist 0.32.0のversion付き公式installerをTLS経由で直接実行し、installer byte列のdigestをrepository契約へ
含めていなかった。公式Releaseはarchive checksumとGitHub Artifact Attestationを提供するが、installerと取得binaryの
検証を各jobで再実装するとversion、platformおよびdigestの管理点が増える。

repositoryの`flake.lock`はcargo-distを含むNix closureを固定済みであるため、CIも同じclosureを使う方がlocal gateとの
同一性を強くできる。従来の`release-workflow-policy.mjs`は文字列検査であり、required fieldが正しいjob、permission scope、
dependency edgeに属することを証明していなかった。

== 実装

* `run-pinned-dist.sh`は`dist-workspace.toml`から期待versionを読み、`flake.lock`で固定されたNix storeの`dist`だけを実行する。
  planとnative buildは同じentry pointを使い、networkから取得したinstallerをshellへ直接渡さない。
* workflowを実行する前にfull commit SHAへ固定したDeterminate Nix actionを使う。native Rust toolchainとmusl toolはrunner側を
  維持し、配布engineだけをlocked closureから解決する。
* `yq`を開発shellへ追加し、policy checkerはYAMLをJSON objectへ変換してjob、`if`、`needs`、permission、environment、
  timeout、retention、action pinおよびstep所属を検証する。`actionlint`はschemaと式検査を引き続き担当する。
* comment、同名文字列、dependency edge削除、unprotected environmentおよびnetwork installerへの逆戻りをnegative testで
  拒否する。
* `yq`は`flake.lock`のNix closureで固定するCI専用toolであり、公開archiveへ同梱しない。このため配布物SBOMと
  `THIRD_PARTY_NOTICES.adoc`のdependency集合には加えず、Issue 104のCI tool監査対象に含める。

== 完了条件

CI bootstrapの実行byte列またはNix closureがversion以外の完全性情報でも固定され、workflow policyがYAML構造に対して
検証される。既存の最小権限、event model、tag-main一致およびtransactional publishのnegative testを維持する。
