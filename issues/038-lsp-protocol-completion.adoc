= Language Serverプロトコル完成

== 目的

標準構文とホスト提供参照解決を利用するLSPの同期、並行処理、設定および参照機能を
完成させる。

== 実装範囲

- Issue 045で確立した`async-lsp`の逐次通知、並行要求、有界CPU解析および
  キャンセル伝播の基盤を再利用し、機能ごとに別の実行経路を作らない。
- workspace configuration、リクエストキャンセル、debounceおよび文書ごとの古いタスク破棄を実装する。
- 複数文書の並行解析上限と異常終了後のホスト側再起動方針を定義する。
- 属性、アンカー、通常リンク、標準cross reference、任意スキーム、言語名および数式に対する
  ホバー・補完対象を定義する。
- `textDocument/definition`、`textDocument/references`および`textDocument/documentLink`
  を実装し、文書・任意スキーム・アンカー解決をホスト注入Resolverへ委譲する。
- アンカー名変更に伴うrenameは、全参照を完全に列挙できるResolverがある場合だけ
  `WorkspaceEdit`を返し、不完全な索引で破壊的編集を行わない。
- 構文ハイライト用semantic tokensをCST・ASTから生成する。
- 単一文書機能と将来のワークスペース索引を別レイヤーにする。
- push diagnostics継続またはpull diagnostics移行を決定し、capabilitiesとfixtureを更新する。

== 完了条件

- キャンセル済みまたは古い解析から診断、補完、Editを返さない。
- position encoding、full・incremental同期およびCRLFの組合せをテストできる。
- Resolver障害時もコア構文機能とサーバプロセスが継続する。
- 文書内、文書間および任意スキーム参照について定義移動と参照検索の位置がUTF-8・UTF-16の
  双方で一致する。
- 未解決参照、曖昧な参照および索引更新中の参照に対して、古い移動先を返さない。
- 同じASTからCLI、ブラウザおよびLSPで一致するシンボルと診断を得られる。

== 実装結果

Issue 045のRouterと有界CPU実行経路へincremental同期、設定可能なdebounce、Definition、
References、Document Link、条件付きRenameおよびSemantic Tokensを追加した。
同一`didChange`内の変更を交渉済みUTF-8・UTF-16位置で逐次適用し、CRLFも共通`LineIndex`で
扱う。診断はgeneration確認付きpush方式を継続する。

open文書は同じ`DocumentStore`の所有`Analysis`から索引する。任意スキームと未open文書は
`HostReferenceIndex`を`run_with_host_index`へ注入して解決する。要求にはsource version、
generation、`ReferenceKey`およびposition encodingを含め、ホストが古い索引結果を拒否
できる。Renameはホスト索引が完全性を宣言した場合だけ返す。

文書属性、アンカー、通常リンク、xrefおよび数式のHover、source言語と文書内アンカーの
補完対象を定義した。Semantic TokensはASTとprojectionから生成し、別parserを持たない。
in-memory protocol、Resolver障害、UTF-8・UTF-16、full・incremental、CRLF、キャンセル、
Definition、ReferencesおよびDocument Linkを自動試験する。

M9完了後の設計レビューでは、`HostReferenceRequest`へsource versionとgenerationを必須化し、
stale索引結果をホスト側でも判定可能にした。外部へはrunnerとホスト索引契約だけを公開し、
文書storeと意味処理serviceは実装詳細へ戻した。

自動完了条件は満たした。Linux版ZedでのDefinition、References、Document Link、
Semantic Tokensおよびincremental同期の回帰確認を残す。
