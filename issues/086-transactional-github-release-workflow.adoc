= TransactionalなGitHub Release workflow
:status: in-progress
:depends-on: 083,084,085

== 目的

検証済みtagの同一commitから全対応targetのassetをbuildし、すべて揃った場合だけGitHub Releaseを公開する。
一部assetだけのreleaseや、再実行による既存assetの上書きを防ぐ。

== 作業

* PRではrelease plan、設定同期および必要に応じたLinux全target buildを副作用なしで実行する。
* tag workflowは最初に`cargo make release-gate`、clean tree、tag／version／contract同期を検査する。
* target別build jobとbrowser library jobはartifactを一時保存し、publish権限を持たない。
* 集約jobはasset集合、重複名、manifest、checksum、SBOM、smoke結果およびprovenance対象を検査する。
* publish jobだけがprotected GitHub Environmentを使い、releaseを作成して全assetをuploadする。
  prerelease tagはGitHub prerelease、stable tagはstable releaseとして扱う。
* 同じtagへの並行実行をconcurrency groupで直列化し、既存releaseまたはassetがある場合は上書きせず失敗する。
* workflow failure時は公開Releaseとpartial assetを残さない。GitHubがReleaseとは独立に保存するattestationを含む、
  公開後の撤回、yank、security incidentおよび削除のrunbookを文書化する。
* repository所有のworkflowをpolicy testで固定する。`dist`はexact versionのplan／build engineとして用い、
  意図的なCI override以外の設定差異をrelease contractで拒否する。

== 完了条件

test repositoryまたはprerelease tagでLinux 2 targetのCLI／LSP、browser library、manifest、checksum、SBOMおよび
attestationが一つのGitHub Releaseに揃う。任意のbuild失敗時にpartial releaseが公開されない。

== 実装状況

`.github/workflows/release.yml`はPRとtagでread-onlyのplan、2 target build、native smoke、browser／metadata build、
5 archiveからのmetadata生成と完全candidate検証までを実行する。tagだけがprotected `github-release` Environmentの
`release-publish.yml`を呼び、private draftへの一括upload、asset集合検査、attestationおよび最後の公開を行う。
同一refのrunは直列化し、既存releaseを拒否する。失敗時のdraft cleanupと公開後のrunbookは
xref:../docs/release-workflow.adoc[release-workflow.adoc]に記載した。

残る完了判定は、GitHubに`github-release` Environmentを設定し、RC tagで2 targetを含む実Releaseと失敗時の
partial release非公開を確認することである。
