= インライン認識とsemantic node構築の分離

== 目的

長大な`parse_segment`と`parse_macro`へ構文追加が集中する状態を解消し、今後のインライン
互換性拡張を局所的かつ検証可能にする。

== 実装範囲

* opener探索、境界判定、閉じ記号探索およびescape判定を副作用のないrecognizerへ分離する。
* recognizerの結果をsemantic `Inline`へ変換するbuilderを分離する。
* link、xref、属性参照およびSTEM macroの共通走査制御と個別payload解析を分離する。
* 回復時の再開位置を型で表し、同じbyteを無制限に再走査しないことを保証する。
* 現在のCore Profile 1のAST、診断、HTML、Formatterおよびprojectionを維持する。

== 完了条件

* 既存の全インライン構文と不完全入力が新しいrecognizer経由で処理される。
* recognizer単体テストが境界、escape、入れ子上限、Unicodeおよび不完全macroを網羅する。
* 1回の走査が入力長に対して有界であることをproperty testまたは計測可能な不変条件で示す。
* nativeとWASMのconformance fixtureが一致する。
* `cargo make release-check`が成功する。

== 依存関係

* 046

== 進捗

macroとmarkerをsource位置順に統合する`InlineCandidate` recognizerを追加し、主走査ループから
候補優先順位とUnicode安全なskip位置の計算を分離した。次にmacro payload認識とsemantic
node構築を分離する。

formula、短縮xref、`xref:`、`link:`およびURLの構文範囲を`MacroToken`として認識する
`recognize_macro`と、再帰的なlabel解析を含む`build_macro`へ分離した。recognizer単体で
各rangeを検査し、不完全入力のskipが必ず次のUTF-8境界へ進み、文字数を超える回数にならない
不変条件を追加した。

通常markerも`MarkerRecognition`と`MarkerToken`でcomplete、invalid、unclosedを分類し、
`build_marker`へsemantic構築を分離した。macro builderはformula、referenceおよびlinkの
小さい処理へ分割し、追加Clippy検査でインライン実装の`too_many_lines`警告がなくなった。

既定設定の`cargo make release-check`は2026-07-20に成功した。Core 153 tests、実WASMとの
9件の適合性比較、3 target各5秒のfuzz smokeを含む。本Issueは完了した。
