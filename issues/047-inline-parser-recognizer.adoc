= インライン認識とsemantic node構築の分離

== 目的

長大な`parse_segment`と`parse_macro`へ構文追加が集中する状態を解消し、今後のインライン
互換性拡張を局所的かつ検証可能にする。

== 実装範囲

* opener探索、境界判定、閉じ記号探索およびescape判定を副作用のないrecognizerへ分離する。
* recognizerの結果をsemantic `Inline`へ変換するbuilderを分離する。
* link、xref、属性参照およびSTEM macroの共通走査制御と個別payload解析を分離する。
* 回復時の再開位置を型で表し、同じbyteを無制限に再走査しないことを保証する。
* macro builderはsemantic nodeだけでなく、再帰的に解析したlabelの診断も呼び出し元へ返す。
* 完全macroと不完全macroを同じ走査器で分類し、後段のheuristicな再走査を削除する。
* 現在のCore Profile 1のAST、診断、HTML、Formatterおよびprojectionを維持する。

== 完了条件

* 既存の全インライン構文と不完全入力が新しいrecognizer経由で処理される。
* recognizer単体テストが境界、escape、入れ子上限、Unicodeおよび不完全macroを網羅する。
* 文字検査回数を計測できる走査器を用い、構文候補が密集した不正入力でも入力byte数に対して
  線形の上限を持つことをproperty testで示す。
* macro label内の不完全書式、入れ子上限および不完全macroが外側の診断へ失われず伝播する。
* nativeとWASMのconformance fixtureが一致する。
* 少なくとも一つの包括的fixtureについて、nativeとWASMの双方から独立した期待AST、診断、
  projectionおよびHTMLを比較する。
* `cargo make release-check`が成功する。

== 進捗

macroとmarkerをsource位置順に統合する`InlineCandidate` recognizerを追加し、主走査ループから
候補優先順位とUnicode安全なskip位置の計算を分離した。次にmacro payload認識とsemantic
node構築を分離する。

formula、短縮xref、`xref:`、`link:`およびURLの構文範囲を`MacroToken`として認識する
`recognize_macro`と、再帰的なlabel解析を含む`build_macro`へ分離した。recognizer単体で
各rangeを検査し、不完全入力のskipが必ず次のUTF-8境界へ進み、文字数を超える回数にならない
不変条件を追加した。

通常markerも`MarkerRecognition`と`MarkerToken`でcomplete、invalid、unclosedを分類し、
`build_marker`へsemantic構築を分離した。macro builderはformula、referenceおよびlinkの
小さい処理へ分割し、追加Clippy検査でインライン実装の`too_many_lines`警告がなくなった。

既定設定の`cargo make release-check`は2026-07-20に成功した。Core 153 tests、実WASMとの
9件の適合性比較、3 target各5秒のfuzz smokeを含む。

その後の批判的レビューで、従来の回復テストはcursorの前進だけを検査し、候補探索がsuffixを
再走査する実装の文字検査量を証明していないことが判明した。また、link、xrefおよび短縮xrefの
labelを再帰解析した際に`InlineParseOutput.problems`を破棄していた。builderの共通結果型で
label診断を伝播し、escaped macroではliteral内容の診断を採用しない修正と回帰試験を追加した。

nativeとWASMの比較はruntime adapter差分を検出するが、同じRustコアに共有された意味バグの
独立なoracleではない。さらに公開ASTへ直接`Serialize`をderiveしたschemaは、semantic APIと
適合性契約を不要に密結合する。Issue 047を再開し、単一走査化、型付きmacro token、明示的な
version付きconformance DTOおよび独立golden fixtureまでを残作業とする。

macro tokenを`FormulaToken`、`ReferenceToken`および`LinkToken`へ分割し、異なる構文familyを
builderへ渡せない型構造へ変更した。これによりreference／link builderのwildcard分岐と
`unreachable!`を削除した。

`MacroRecognition`がcomplete、incompleteおよびinvalidを分類し、不完全macroの診断と
UTF-8安全な再開位置を主走査へ返すようにした。後段の`scan_incomplete_macros`を削除し、
不完全xrefの後ろにある正常なinline構文を引き続き救済する回帰試験を維持した。次の作業として、
候補探索と閉じ記号探索の重複文字検査を一つのscannerへ統合し、計測可能な線形上限を
property testで固定する方針を設定した。

候補列を一回の前進走査で構築し、各候補を一度だけ取り出す`InlineScanner`へ移行した。
markerの有効closerは一回の逆向き走査、macroの`[`、`]`および`>>`は一回のdelimiter走査で
索引化し、recognizerからsuffixの`find`を除去した。URL系macroはtoken境界からだけ認識し、
scheme探索が空白や別tokenへ越境しない規則を規範文法へ追加した。生成した密集不正入力を含め、
scannerの検査量が`2 * Unicode scalar数 + UTF-8 byte数`に一致し、したがって入力byte数の
3倍以下となる不変条件をテストで固定した。
