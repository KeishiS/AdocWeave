= CI実行時間の最適化
:status: in-progress
:depends-on: 132

== 目的

品質、current stable Rust、release candidateおよびpublishの検証責務を維持したまま、GitHub Actionsの待ち時間、
重複buildおよび不要なartifact転送を減らす。

== 作業

* workflow jobごとのwall-clock time、queue time、cache hit率およびartifact sizeを計測する。
* quality、native target build、WASM、Zedおよびrelease artifactの依存graphを見直す。
* Rust、NixおよびNode cacheのkey・復元範囲を、lockfileとtargetに基づき再設計する。
* security、current stable Rust、cross-runtime、release contractおよび公開前の完全性検証を省略しないことをpolicy testで固定する。

== 完了条件

* 現在の必須検証を保持した比較可能な計測結果と、採用した最適化の根拠を記録する。
* fork、pull request、main、tagの権限境界とartifact provenanceを弱めない。

== 計測基準

測定対象はGitHub Actionsのjob APIが返す`created_at`、`started_at`および`completed_at`とする。queue timeは
`started_at - created_at`、wall-clock timeは`completed_at - started_at`で定義する。cache hit率は、明示的な
cache actionが存在せずNix binary cacheのsubstitute結果もActions APIから取得できないため、このCIの最適化指標には
含めない。代わりにlockfile固定とjobの依存edgeを再現可能な最適化単位とする。

== 採用した最適化

* source quality、fuzz、pull requestのNix package検証を独立jobに分離した。
* mainおよびtagではcandidate buildをplan直後に開始し、publishだけがquality、fuzz、candidate verificationおよび
  installation E2Eの全成功を待つ。
* mainおよびtagで重複していたx86-64のNix package buildを省き、candidate installation E2EのLinux 2 architectureで
  同じpackage検証を実施する。
* native release buildをNix development shellで実行し、runnerの`rustup`ではなく、release manifestと一致する
  lockfile固定Rust toolchainを使用する。
