= AdocWeave 実装Issue一覧

各Issueは、単独で動作確認できる最小限の変更単位を表す。原則として番号順に進めるが、依存関係を満たすIssueは並行して実施してよい。

本計画は標準AsciiDocの部分集合とホスト非依存な解析・出力・エディター基盤を扱う。
特定ノートアプリの属性スキーマ、DB投影、サーバおよび製品統合は外部リポジトリが所有する。

== 運用方針

- 一つのIssueを、可能な限り一つの独立したコミットとして完了する。
- 構文追加では、解析だけでなくHTML、診断、LSP、整形のうち該当する機能まで縦に実装する。
- fixtureを用意し、正常系、不完全入力、境界値を確認する。
- 各Issueの完了時に `cargo fmt`、`cargo clippy`、関連テストを実行する。
- 未対応構文を推測して変換せず、安全なプレーンテキストまたは診断として扱う。
- 文法、置換、HTML出力または公開APIを変更するIssueでは、対応する規範的仕様と互換性fixtureも同じIssueで更新する。
- ネイティブ、WASM、CLI、LSPおよびサーバ統合で別々の簡易パーサーを実装しない。
- semantic ASTと解析APIへHTML固有のタグ、属性、classまたは出力設定を持ち込まない。各出力形式は独立したバックエンドとして同じASTを読み取る。
- semantic ASTでは、内容を再解釈しないリテラルノードと、子要素へ置換を適用するコンテナノードを区別する。出力バックエンドはこの意味上の区別を各形式の表現へ写像する。
- CLI等のフロントエンドは解析と出力バックエンドを組み合わせるだけとし、レンダラー内で再解析しない。
- 公開APIはファイル、ネットワーク、環境変数、時刻、UUID解決等のホスト機能へ暗黙にアクセスしない。
- cross referenceは構文解析、対象索引、ホスト解決および最終URL生成を分離する。
  コアは未解決参照を保持するが、ファイルシステムやDBへアクセスしない。
- 文書内、文書間および任意スキーム参照は共通の参照型、診断、投影およびLSP問い合わせを
  利用し、通常の外部URLとは型で区別する。
- リポジトリの説明文書は原則として日本語で記述し、形式はAsciiDoc（`.adoc`）に統一する。
- Issue番号は依存順を示す。実装開始前に各Issueの未決事項が解消されていることを確認する。

=== M12: 持続可能な解析基盤

Issue 001から051の実装を対象とした批判的レビューに基づき、構文追加より先に
安全性、構文表現および公開snapshotの不変条件を再構築する。この段階では
後方互換性を要求せず、影響する契約versionとfixtureを各Issueで更新する。

M12は全体を一つの破壊的移行として扱う。各Issueは設計・差分レビューの境界を示すが、
Issue単位でworkspace全体がbuild・動作することは要求しない。旧CST・ASTと新syntax・semantic
modelを長期間並存させるadapterは作らず、最後のIssue 058完了時にnative、WASM、CLI、LSP、
Zed連携、文書および全検証gateが一斉に動作することを完了条件とする。ただし各途中commitは
対象範囲の不変条件を可能な限りtestで固定し、未完了状態と既知の失敗をIssueへ明記する。
