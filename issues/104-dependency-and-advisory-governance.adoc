= 依存更新、licenseおよびadvisory governance
:status: completed
:depends-on: 103

== 目的

root workspace、独立Zed crate、browser packageおよびCI toolの依存更新を一つの監査手順で扱い、既知脆弱性、MSRV、
license、重複versionおよびrelease SBOMの差異を継続的に検出する。

== 実装

* `dependency-boundaries.json`でroot、Zed、browser、Nixおよびfuzzのmanifest／lockfileと配布境界を列挙した。
* Nixで固定したcargo-audit 0.22.2とcargo-deny 0.20.2を採用した。RustSec DBは40桁revisionへ固定し、取得失敗とHEAD不一致を
  hard failureにする。
* SPDX licenseとcrates.io source allowlistを両Cargo graphへ適用し、noticeの不足だけでなく余分な旧packageも拒否する。
  release SBOMは同じCargo metadata集合とfrontend manifestから生成する。
* 実在する複数versionだけを理由付きbaselineへ固定し、増加と解消の両方をreview対象にした。
* rootとZedのMSRVをRust 1.86へ統一し、qualityの独立jobでadvisory、license、source、重複、noticeおよび境界を検査する。
* 例外registryはowner、理由、期限およびrepository Issueを必須とし、期限切れと包括ignoreを拒否する。
* 自動更新serviceはRC安定化中には採用せず、月次のDB／依存review手順を
  xref:../docs/dependency-governance.adoc[依存関係governance]へ定義した。

== 完了条件

全配布境界の依存集合が一つの監査結果へ統合され、advisory、license、MSRV、noticeおよびSBOMの不一致がCIで失敗する。
例外はowner、理由、期限および追跡Issueを持ち、無期限の包括ignoreを認めない。
